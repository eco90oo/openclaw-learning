## 1️⃣ 題目說明

### 面試官會怎麼問？

> 「請設計一個集中的日誌收集與分析系統，需要能收集所有伺服器的日誌、支援全文檢索、並能快速查詢過去 30 天的資料。」

這題考察候選人對**可觀測性 (Observability)** 的理解，特別是日誌管理這個領域。常見的類似問題還有「設計 ELK Stack」或「設計分散式日誌系統」。

### 需要澄清什麼問題？

在開始畫圖之前，你需要問面試官以下問題：

1. **日誌來源**：要收集哪些類型的日誌？（應用日誌、系統日誌、存取日誌、審計日誌）
2. **傳輸方式**：日誌要怎麼傳到中央系統？（Push 還是 Pull？）
3. **儲存需求**：日誌要保留多久？需要支援多長的查詢範圍？（7天？30天？1年？）
4. **查詢需求**：需要支援哪些查詢場景？（關鍵字搜尋、複雜過濾、聚合分析）
5. **規模考量**：每秒多少日誌量？峰值多少？（1GB/s？10GB/s？）

## 2️⃣ 思考框架

### Step 1: 需求釐清

先問清楚這些問題，然後在心裡有個初步的輪廓：

- **功能需求**：收集 → 傳輸 → 儲存 → 索引 → 查詢 → 可視化
- **非功能需求**：高可用、低延遲、可擴展、支援全文檢索
- **規模假設**：假設每秒收集 10GB 日誌，保留 30 天，總共約 25PB

## 3️⃣ 詳細解答

### 3.1 為什麼需要日誌收集系統？

**比喻**：想像你是公司 IT 部門的主管，有 1000 台伺服器。如果每台伺服器出問題，你都要遠端登入去查看日誌 - 這就像是要去 1000 個不同的文件櫃找資料，太辛苦了！

集中的日誌收集系統就像建立了一個**中央圖書館**：
- 所有伺服器的日誌都自動送到這裡
- 你可以用關鍵字快速搜尋
- 可以看到整個系統的健康狀況

### 3.4 程式碼範例

#### A. Filebeat 配置範例

```yaml
# filebeat.yml
filebeat.inputs:
  - type: log
    enabled: true
    paths:
      - /var/log/myapp/*.log
    fields:
      service: myapp
      environment: production
    json.keys_under_root: true
    json.overwrite_keys: true

output.kafka:
  hosts: ["kafka-1:9092", "kafka-2:9092", "kafka-3:9092"]
  topic: "logs-%{[fields.service]}"
  partition.hash:
    hash:
      - field

processors:
  - add_host_metadata:
      when.not.contains.tags: forwarded
  - add_docker_metadata: ~
```

#### B. Logstash ETL 處理

```ruby
# logstash pipeline.conf
input {
  kafka {
    bootstrap_servers => "kafka:9092"
    topics => ["logs-myapp"]
    group_id => "logstash-processor"
  }
}

filter {
  # 解析時間戳
  date {
    match => ["timestamp", "ISO8601"]
    target => "@timestamp"
  }

  # 添加來源資訊
  mutate {
    add_field => {
      "source_zone" => "%{[fields][zone]}"
      "environment" => "%{[fields][environment]}"
    }
  }

  # 過濾敏感資訊
  mutate {
    gsub => [
      "message", "[0-9]{16,19}", "[REDACTED]"
    ]
  }
}

output {
  elasticsearch {
    hosts => ["elasticsearch:9200"]
    index => "logs-%{+YYYY.MM.dd}"
  }
}
```

#### C. Elasticsearch 索引模板

```json
{

  "template": {

      "number_of_shards": 3,

        "trace_id": { "type": "keyword" },
```

## 4️⃣ 延伸問題

### 面試官可能會追問的問題：

| 問題 | 解答要點 |
|------|----------|
| **如何處理大日誌量？** | 使用消息佇列緩衝、分區處理、壓縮傳輸 |
| **如何確保日誌不丟失？** | 至少一次傳遞 (at-least-once)、副本機制、持久化 |
| **如何控制成本？** | 分層儲存 (Hot/Warm/Cold)、壓縮、Lifecycle policies |
| **如何保證安全性？** | TLS 傳輸加密、角色權限控制、日誌脫敏 |
| **如何處理延遲？** | 優化批次大小、選擇合適的副本數、硬體優化 |

### 進階追問：

1. **如何設計多租戶隔離？**
   - 使用 Index 隔離每個租戶
   - 使用 Search Guard 做權限控制

2. **如何做日誌歸檔？**
   - 使用 ILM (Index Lifecycle Management)
   - Hot → Warm → Cold → Delete

3. **如何處理結構化 vs 非結構化日誌？**
   - 結構化：JSON → 直接解析
   - 非結構化：Grok Pattern 解析

# 📚 【今日學習：設計 Google Docs/協作系統】

## 1️⃣ 題目說明

### 面試官會怎麼問？

「請設計一個像 Google Docs 的多人協作文字編輯系統。用戶可以建立文件、邀請他人共同編輯、即時看到彼此的修改（類似 Google Docs 的協作功能）。請說明你會如何設計這個系統，確保多人同時編輯時不會發生衝突，並提供流暢的使用體驗。」

### 需要澄清什麼問題？

在動手畫圖之前，先問清楚這些：

1. **流量規模**：
   - 同時編輯人數上限？（2 人？10 人？100 人？）
   - 文件數量規模？（個人使用？企業級？）
   - 每日活躍用戶（DAU）是多少？

2. **功能範圍**：
   - 只需要文字編輯？還是也要支援表格、簡報？
   - 需要離線編輯功能嗎？
   - 需要版本歷史（Revision History）嗎？
   - 需要權限管理（誰可以編輯、誰只能檢視）？

3. **協作需求**：
   - 是否需要即時通訊功能（Comments、Chat）？
   - 需要支援多少種文件格式（.docx、.pdf、.md）？
   - 是否需要 Export/Import 功能？

4. **服務品質**：
   - 編輯延遲要求？（輸入到看到更新的延遲）
   - 離線修改如何處理？（重新上線後同步）
   - 資料隱私與安全性要求？

### Step 3: 資料庫設計

```sql
-- 用戶表
CREATE TABLE users (
    id UUID PRIMARY KEY,
    email VARCHAR(255) UNIQUE NOT NULL,
    name VARCHAR(255),
    password_hash VARCHAR(255),
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);

-- 文件表
CREATE TABLE documents (
    id UUID PRIMARY KEY,
    title VARCHAR(255) NOT NULL,
    content TEXT,  -- 完整內容快照
    owner_id UUID REFERENCES users(id),
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW(),
    version BIGINT DEFAULT 0  -- 用於樂觀鎖
);

-- 文件權限表
CREATE TABLE document_permissions (
    id UUID PRIMARY KEY,
    document_id UUID REFERENCES documents(id),
    user_id UUID REFERENCES users(id),
    permission ENUM('view', 'comment', 'edit') NOT NULL,
    created_at TIMESTAMP DEFAULT NOW(),
    UNIQUE(document_id, user_id)
);

-- 操作日誌表（用於版本歷史）
CREATE TABLE operations (
    id UUID PRIMARY KEY,
    document_id UUID REFERENCES documents(id),
    user_id UUID REFERENCES users(id),
    operation_type VARCHAR(50),  -- 'insert', 'delete', 'format'
    operation_data JSONB,        -- 操作的具體內容
    version BIGINT,               -- 操作後的版本號
    created_at TIMESTAMP DEFAULT NOW()
);

-- 版本快照表（定期保存完整內容）
CREATE TABLE document_snapshots (
    id UUID PRIMARY KEY,
    document_id UUID REFERENCES documents(id),
    content TEXT,
    version BIGINT,
    created_at TIMESTAMP DEFAULT NOW()
);
```

### 🔑 核心概念：Operational Transformation (OT)

讓我用一個生活化的比喻來解釋：

> **比喻：兩個人一起寫白板**
> 
> 想象你和同事一起在白板上寫字。你在左邊寫「Hello」，同事在右邊寫「World」。你們同時寫，不會衝突。
> 
> 但如果你們同時修改同一個地方呢？比如你想把「Hello」改成「Hi」，同時同事想改成「Hey」。這就是**衝突問題**。

**OT 的核心思想**：
- 每次編輯操作都被記錄為一個「操作」（Operation）
- 當多個操作同時發生時，用數學方法將它們「轉換」成不會衝突的版本
- 伺服器擔任「仲裁者」，決定操作的順序

```python
# 簡化的 OT 操作類型
class Operation:
    def __init__(self, op_type, position, text):
        self.type = op_type      # 'insert' 或 'delete'
        self.position = position # 操作位置
        self.text = text         # 插入/刪除的文字
    
    # 轉換函數：合併兩個並發操作
    @staticmethod
    def transform(op1, op2):
        """
        當兩個操作同時發生時，如何合併？
        """
        if op1.position < op2.position:
            # op1 在前面，op2 的位置不需要調整
            return op1, op2
        elif op1.position > op2.position:
            # op1 在後面，需要往後移動
            adjusted_op1 = Operation(
                op1.type,
                op1.position + len(op2.text),
                op1.text
            )
            return adjusted_op2, op1
        else:
            # 位置相同，需要人為決定順序
            # （通常根據用戶 ID 排序）
            return op1, op2
```

### 🔑 替代方案：CRDT (Conflict-free Replicated Data Types)

> **比喻：Google Docs 的「建議修改」模式**
> 
> 有些協作系統不是直接修改，而是「建議」改變。每個建議都是獨立的，最終用戶可以選擇接受或拒絕。
> 
> CRDT 也是類似的概念：每個操作都是「可合併」的，不需要中央仲裁者。

```python
# 簡化的 CRDT 字串实现
class CRDTString:
    """
    使用 Position-based Set 實現衝突自由字串
    每個字元都有唯一 ID，不會衝突
    """
    def __init__(self):
        self.characters = []  # (id, char, user_id)
    
    def insert(self, position, char, user_id):
        # 產生唯一 ID
        unique_id = self._generate_id(user_id)
        self.characters.insert(position, (unique_id, char, user_id))
        self._sort()
    
    def _generate_id(self, user_id):
        # 簡化的 ID 生成（實際使用更複雜的邏輯）
        import time
        import random
        return f"{time.time()}-{user_id}-{random.randint(0, 999)}"
    
    def _sort(self):
        # 按 ID 排序以保持一致
        self.characters.sort(key=lambda x: x[0])
    
    def merge(self, other):
        # 合併兩個 CRDT 字串
        all_chars = self.characters + other.characters
        # 去重並排序
        unique_chars = {}
        for char_id, char, user_id in all_chars:
            if char_id not in unique_chars:
                unique_chars[char_id] = (char_id, char, user_id)
        self.characters = sorted(unique_chars.values(), key=lambda x: x[0])
```

## 4️⃣ 延伸問題

### ❓ 面試官可能會追問什麼？

1. **「如果網路斷線了，用戶的編輯會不會丟失？」**
   - 解答：需要 Client-side 緩存，離線時本地存儲，重新連線後同步

2. **「如何處理同時 100 人編輯同一個文件？」**
   - 解答：考慮分片（Shard）、使用 CRDT 而非 OT，或限制同時編輯人數

3. **「如何實現 Google Docs 的『建議修改』功能？」**
   - 解答：新增「建議」類型的操作，需要 Owner 審核通過才正式生效

4. **「如何實現Undo/Redo功能？」**
   - 解答：在 Client 端維護操作歷史棧，伺服器端保存完整操作日誌

5. **「如何確保資料隱私？文件內容加密？」**
   - 解答：傳輸加密（TLS）、儲存加密（AES），可選客戶端加密（E2EE）

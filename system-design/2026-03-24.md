# 📚 【今日學習：設計 YouTube/Netflix 影片串流系統】

## 1️⃣ 題目說明

### 面試官會怎麼問？

「請設計一個像 YouTube 或 Netflix 的影片串流服務。用戶可以上傳影片、觀看影片、搜尋影片，並且影片應該能夠流暢播放，特別是在不同網路環境下（4G、WiFi、弱網）。請說明你會如何設計這個系統，讓數億用戶能夠同時觀看影片。」

### 需要澄清什麼問題？

在動手畫圖之前，先問清楚這些：

1. **流量規模**：
   - 有多少用戶？每日活躍用戶（DAU）是多少？
   - 每天有多少新影片上傳？總影片庫存有多少小時？
   - 同時有多少人在觀看影片？（峰值並發）

2. **功能範圍**：
   - 只需要基本的上傳、播放、搜尋？
   - 還需要「評論」、「按讚」、「訂閱」、「建議影片」？
   - 是否需要支援「直播」？

3. **影片特性**：
   - 影片長度限制？（短影片 1 分鐘 vs 電影 2 小時）
   - 影片解析度需求？（720p、1080p、4K、HDR）
   - 是否需要離線下載功能？

4. **服務品質**：
   - 影片加载時間要求？（首次緩衝要多久）
   - 播放失敗率要求？（成功率要多少個 9）
   - 是否需要支援「倍速播放」？

## 3️⃣ 詳細解答

### 3.1 上傳與轉碼流程（最核心！）

```
┌─────────────────────────────────────────────────────────────────────┐
│                     上傳 + 轉碼 完整流程                            │
└─────────────────────────────────────────────────────────────────────┘

   用戶端                    後端服務                  轉碼集群              儲存
     │                         │                       │                    │
     │  1. POST /upload       │                       │                    │
     │───────────────────────>│                       │                    │
     │                         │                       │                    │
     │                    2. 建立記錄                 │                    │
     │                    Generate upload_id         │                    │
     │                         │                       │                    │
     │  3. 返回 upload_url    │                       │                    │
     │<───────────────────────│                       │                    │
     │                         │                       │                    │

     │     儲存服務           │                       │                    │
     │───────────────────────────────────────────────>│                    │
     │                         │                       │                    │
     │                    5. 上傳完成                 │                    │
     │                    POST /complete              │                    │
     │───────────────────────>│                       │                    │
     │                         │                       │                    │
     │                    6. 發送轉碼任務              │                    │
     │                    到 Message Queue            │                    │
     │───────────────────────>│                       │                    │

     │                         │                       │                    │

     │                         │<──────────────────────│                    │
     │                         │                       │                    │
     │                         │  9. FFmpeg 轉碼      │                    │
     │                         │  (多品質：360p/720p  │                    │
     │                         │   1080p/4K)          │                    │
     │                         │─────────────────────>│                    │
     │                         │                       │                    │
     │                         │  10. 上傳轉碼後影片  │                    │

     │                         │  11. 更新資料庫       │                    │

     │                         │                       │                    │
     │  12. 可播放！           │                       │                    │
     │<───────────────────────│                       │                    │
```

**為什麼要做轉碼？**

想像一下：
- 用戶 A 用 iPhone 13 Pro 拍攝 4K 影片（檔案 500MB）
- 用戶 B 用舊款 Android 手機觀看
- 用戶 C 用 4K 電視觀看
- 用戶 D 在捷運上用 4G 網路觀看

如果不轉碼，會發生什麼事？
- 4K 影片在舊手機上跑不動
- 500MB 檔案用 4G 網路根本無法載入
- 每個影片都要下載完整，太浪費頻寬

**轉碼就是「因材施教」**：根據不同裝置和網路，提供最適合的影片品質！

**比喻**：就像自助餐！
- 傳統下載 = 把整盤菜端走（浪費）
- 串流 = 吃多少夾多少（剛剛好）

## 4️⃣ 延伸問題

面試官可能會追問的問題：

### 4.1 如果影片上傳失敗了怎麼辦？

```
┌─────────────────────────────────────────────────────────────────┐
│                        斷點續傳 (Resume Upload)                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  問題：如果用戶上傳 1GB 影片，網路斷了，要從頭來過嗎？             │
│                                                                 │
│  解決：支援分塊上傳 + 斷點續傳                                   │
│                                                                 │
│  1. 將影片切成多個 chunk (如 5MB)                               │
│  2. 每個 chunk 獨立上傳                                         │
│  3. 客戶端紀錄上傳到哪個 chunk                                  │
│  4. 網路斷線後，重新連線從斷點繼續                               │
│                                                                 │
│  ┌────────────────────────────────────────────────────────────┐│
│  │  Chunk 1 │ Chunk 2 │ Chunk 3 │ Chunk 4 │ Chunk 5 │ ...  ││
│  │    ✅     │   ✅    │   ❌    │   ⏸️   │ 未上傳  │      ││
│  │                     (網路斷線)                              ││
│  │                     從 Chunk 3 繼續！                        ││
│  └────────────────────────────────────────────────────────────┘│
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 4.2 如何防止盜版/盜連？

```
┌─────────────────────────────────────────────────────────────────┐
│                        防盜連策略                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 簽名 URL (Signed URL)                                      │
│     • 只有帶正確簽名的 URL才能存取                              │
│     • 過期時間內有效                                            │
│     • 示例：                                                    │
│       https://cdn.example.com/video.mp4?                        │
│         signature=abc123&expires=1700000000                    │
│                                                                 │
│  2. Referer 檢查                                               │
│     • 只允許來自自家網站的請求                                   │
│     • 但容易被繞過                                            │
│                                                                 │
│  3. DRM (Digital Rights Management)                            │
│     • 用於付費內容                                              │
│     • Widevine/ FairPlay/ PlayReady                           │
│                                                                 │
│  比喻：電影票                                                    │
│     只能看一次 + 指定座位 + 進場驗票                            │
│     不能把票給別人讓他進場                                       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 4.3 如何處理熱門影片（病毒式傳播）？

```
┌─────────────────────────────────────────────────────────────────┐
│                     熱門影片的挑戰                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  情境：一個影片突然爆紅，100萬人同時觀看                        │
│                                                                 │
│  問題：CDN 撐得住嗎？原始伺服器會不會掛掉？                     │
│                                                                 │
│  解決策略：                                                     │
│                                                                 │
│  1. 預先載入 (Pre-warming)                                     │
│     • 偵測即將熱門的影片                                        │
│     • 提前將影片推送到 CDN 邊緣節點                             │
│                                                                 │
│  2. 請求層級限流                                                │
│     • 使用 Rate Limiter                                        │
│     • 過濾異常流量                                              │
│                                                                 │
│  3. 多層級快取                                                  │
│     • L1: CDN edge nodes                                       │
│     • L2: CDN origin shields                                   │
│     • L3: 自己的快取伺服器                                     │
│                                                                 │
│  4. 灰度發布                                                    │
│     • 先讓少部分人看                                            │
│     • 逐步放量                                                  │
│                                                                 │
│  比喻：餐廳大胃王比賽                                           │
│      平常 100 人吃飯                                            │
│      比賽時 10000 人來                                          │
│      要提前準備 + 分流接待 + 廚房擴容                           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 4.4 影片編碼格式有哪些？

```
┌─────────────────────────────────────────────────────────────────┐
│                     常見影片編碼格式                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  🎬 容器格式 (Container)                                       │
│     • MP4 - 最常見，支援廣                                       │
│     • MKV - 開放格式，支援多音軌                                │
│     • WebM - 為 Web 設計，Google 推廣                           │
│                                                                 │
│  🎥 視訊編碼 (Video Codec)                                     │
│     • H.264 - 最普及，相容性好                                  │
│     • H.265 (HEVC) - 更好的壓縮率，但專利                       │
│     • VP9 - Google 開源，免專利費                               │
│     • AV1 - 最新最強，開源免費                                  │
│                                                                 │
│  🔊 音訊編碼 (Audio Codec)                                     │
│     • AAC - MP4 最常用                                          │
│     • Opus - 壓縮率高，VoIP 適合                                │
│                                                                 │
│  壓縮效率比較：                                                 │
│     H.264  >  VP9  >  H.265  >  AV1                            │
│     (同樣畫質，檔案大小)                                        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 5️⃣ 類似題目

相關的系統設計問題：

1. **設計直播系統**
   - 需要即時性 → WebSocket/RTMP
   - 延遲要求更高 → < 3 秒
   - 需要即時互動功能

2. **設計音樂串流服務（Spotify）**
   - 和影片類似，但檔案更小
   - 需要離線播放
   - 歌單管理功能

3. **設計 Podcast 平台**
   - 需要 RSS 訂閱功能
   - 下載管理
   - 播放進度同步

4. **設計檔案分享服務（Dropbox）**
   - 大檔案上傳
   - 版本控制
   - 同步機制

## 📝 重點複習

| 核心概念 | 關鍵重點 |
|---------|---------|
| **轉碼** | 將原始影片轉為多個品質（360p/720p/1080p/4K）|
| **串流** | HLS/DASH 分段串流，客戶端自適應調整品質 |
| **CDN** | 全球分發，就近存取，邊緣快取 |
| **儲存** | 原始影片 → 轉碼 → 分發，分層管理 |
| **擴展** | 水平擴展轉碼 Worker + CDN 全球化 |

# 📚 【今日學習：設計雲端儲存系統（Dropbox/Google Drive）】

## 1️⃣ 題目說明

### 面試官會怎麼問？
> 「請設計一個類似 Dropbox 或 Google Drive 的雲端檔案儲存服務，用戶可以上傳檔案、資料夾管理、分享檔案給其他人、還有檔案版本控制。請告訴我你的整體架構設計。」

### 需要澄清的問題（Step 0: 需求釐清）
在開始畫圖之前，你需要先問清楚這些關鍵問題：

1. **規模多大？**
   - 是小型個人雲端（幾百GB）還是企業級（PB等級）？
   - 同時在線用戶數量級是多少？

2. **功能範圍？**
   - 只需要基本的上傳/下載？
   - 還是要支援線上編輯、協作？

3. **檔案類型？**
   - 小檔案為主（文件、圖片）？
   - 還是有大檔案（影片、設計檔）？

4. **一致性要求？**
   - 需要即時同步？還是可以接受 eventual consistency？

5. **誰來用？**
   - 個人開發者？
   - 企業內部使用？

### Step 3: 資料庫設計

#### 核心資料表設計

**Users 表（用戶）**
```sql
CREATE TABLE users (
    id BIGINT PRIMARY KEY,
    email VARCHAR(255) UNIQUE NOT NULL,
    name VARCHAR(255),
    created_at TIMESTAMP,
    storage_quota BIGINT DEFAULT 15GB
);
```

**Files 表（檔案元資料）**
```sql
CREATE TABLE files (
    id BIGINT PRIMARY KEY,
    user_id BIGINT NOT NULL,
    parent_folder_id BIGINT,  -- NULL = 根目錄
    name VARCHAR(255) NOT NULL,
    file_path VARCHAR(1024),  -- 完整路徑 /user/folder/file.txt
    size BIGINT,               -- 檔案大小
    content_hash VARCHAR(64), -- SHA-256 雜湊值（去重複）
    mime_type VARCHAR(100),
    is_folder BOOLEAN DEFAULT FALSE,
    version INT DEFAULT 1,    -- 版本號
    created_at TIMESTAMP,
    updated_at TIMESTAMP,
    deleted_at TIMESTAMP,     -- 軟刪除
    FOREIGN KEY (user_id) REFERENCES users(id)
);
```

**Shares 表（分享）**
```sql
CREATE TABLE shares (
    id BIGINT PRIMARY KEY,
    file_id BIGINT NOT NULL,
    owner_id BIGINT NOT NULL,
    shared_with_email VARCHAR(255),
    permission VARCHAR(20),  -- 'read', 'write', 'admin'
    share_token VARCHAR(64) UNIQUE,
    expires_at TIMESTAMP,
    created_at TIMESTAMP
);
```

**FileVersions 表（版本控制）**
```sql
CREATE TABLE file_versions (
    id BIGINT PRIMARY KEY,
    file_id BIGINT NOT NULL,
    version INT NOT NULL,
    content_hash VARCHAR(64),
    size BIGINT,
    created_at TIMESTAMP,
    created_by BIGINT
);
```

#### 索引設計
```sql
-- 快速查詢某用戶的根目錄檔案
CREATE INDEX idx_files_user_parent ON files(user_id, parent_folder_id);

-- 查詢某路徑下的所有檔案
CREATE INDEX idx_files_path ON files(user_id, file_path);

-- 查詢某檔案的所有版本
CREATE INDEX idx_versions_file ON file_versions(file_id, version DESC);
```

### Step 5: 擴展性考量

#### 📌 挑戰 1：大檔案上傳
**問題**：影片、設計檔可能好幾GB，HTTP 上傳會失敗

**解決方案：分塊上傳（Chunked Upload）**

```
┌────────────────────────────────────────────────────┐
│              分塊上傳流程                          │
├────────────────────────────────────────────────────┤
│  1. 初始化上傳                                     │
│     POST /api/upload/init                         │
│     → 返回 upload_id                              │
│                                                    │
│  2. 上傳每個分塊（Parallel）                        │
│     PUT /api/upload/{upload_id}/chunk/{index}    │
│     → 每個 chunk 獨立上傳，可並發                   │
│                                                    │
│  3. 完成上傳                                       │
│     POST /api/upload/{upload_id}/complete         │
│     → 合併所有 chunks，產生完整檔案                │
└────────────────────────────────────────────────────┘
```

**比喻**：就像搬家時把所有東西分裝成小箱子，分別搬運，最後再組合成完整家具。

#### 📌 挑戰 2：去重複（Dedup）
**問題**：相同檔案只存一份，節省空間

**解決方案：Content-Addressable Storage**

```
┌────────────────────────────────────────────────────┐
│              去重複機制                           │
├────────────────────────────────────────────────────┤
│                                                    │
│   檔案 A (Alice)                                  │
│   └── SHA-256: a1b2c3...                         │
│        │                                          │
│        ▼                                          │
│   ┌─────────┐                                     │
│   │ 檢查雜湊│ ────► 存在？直接指向現有檔案         │
│   └────┬────┘         不存在？建立新區塊           │
│        │                                          │
│        ▼                                          │
│   ┌─────────────────────────────────────────┐    │
│   │          Object Storage                 │    │
│   │  a1b2c3... → [實際檔案內容]              │    │
│   └─────────────────────────────────────────┘    │
│                                                    │
│   檔案 B (Bob) 上傳相同內容                        │
│   └── SHA-256: a1b2c3... → 直接指向同一區塊！     │
│                                                    │
└────────────────────────────────────────────────────┘
```

**比喻**：就像圖書館的 ISBN 系統，相同ISBN的書只會有一本編目，其他人借閱只是指向同一本書。

#### 📌 挑戰 3：同步引擎
**問題**：用戶在不同裝置上，如何保持同步？

**解決方案：雙向同步 + 事件驅動**

```
┌────────────────────────────────────────────────────┐
│              同步流程                              │
├────────────────────────────────────────────────────┤
│                                                    │
│   Client A (筆電)  ──改變檔案──► Server            │
│                            │                       │
│                            ▼                       │
│                    ┌───────────────┐              │
│                    │ Sync Engine   │              │
│                    │ • 計算差異    │              │
│                    │ • 衝突解決    │              │
│                    └───────┬───────┘              │
│                            │                       │
│         ┌──────────────────┼──────────────────┐   │
│         ▼                  ▼                  ▼   │
│   Client B (手機)   Client C (平板)   Client D   │
│                                                    │
└────────────────────────────────────────────────────┘
```

**Sync Protocol 設計：**
```python
# 客戶端同步演算法
class SyncClient:
    def sync(self):
        # 1. 獲取本地變更
        local_changes = self.get_local_changes()
        
        # 2. 獲取伺服器變更（since last sync）
        remote_changes = self.api.get_changes(
            since=self.last_sync_token
        )
        
        # 3. 合併（處理衝突）
        merged = self.merge(local_changes, remote_changes)
        
        # 4. 衝突處理策略
        # - 最後寫入者勝（Last-Write-Wins）
        # - 詢問用戶
        # - 自動合併（對於純文字）
        
        # 5. 同步到本地
        self.apply_changes(merged)
        
        # 6. 更新 sync token
        self.last_sync_token = remote_changes.new_token
```

### 關鍵元件說明

#### 1. Object Storage（S3/MinIO）
```python
# 檔案上傳流程
async def upload_file(user_id, file_data, metadata):
    # 1. 計算 content hash（用於去重複）
    content_hash = sha256(file_data)
    
    # 2. 檢查是否已存在
    existing = await check_hash_exists(content_hash)
    
    if existing:
        # 已存在，只需建立元資料引用
        storage_key = existing.storage_key
    else:
        # 3. 上傳到 Object Storage
        storage_key = f"{user_id}/{uuid()}"
        await s3.upload(storage_key, file_data)
    
    # 4. 建立檔案元資料
    file_record = await db.create_file(
        user_id=user_id,
        name=metadata.name,
        storage_key=storage_key,
        content_hash=content_hash,
        size=len(file_data)
    )
    
    return file_record
```

#### 2. 同步衝突解決策略

```python
# 衝突解決演算法
class ConflictResolver:
    """
    三種策略：
    1. Last-Write-Wins (LWW): 時間戳記比較
    2. Manual: 讓用戶選擇
    3. Auto-merge: 對於文字檔可以自動合併
    """
    
    @staticmethod
    def last_write_wins(local: FileVersion, remote: FileVersion):
        if local.updated_at > remote.updated_at:
            return local
        return remote
    
    @staticmethod
    def merge_text_files(base: str, local: str, remote: str):
        # 使用三向合併（Three-way merge）
        # 類似 Git 的合併策略
        return merge3.merge(base, local, remote)
```

## 4️⃣ 延伸問題

面試官可能會追問這些問題：

### ❓ Q1: 如何處理大量小檔案？
**答**：使用 Merkle Tree 加速同步
- 類似 Git 的物件模型
- 只同步變更的部分
- 對大量小檔案非常有效

### ❓ Q2: 儲存空間用完了怎麼辦？
**答**：
- 軟性限制：提醒用戶升級
- 硬性限制：禁止上傳新檔案
- 設計收費機制

### ❓ Q3: 如何實作「團隊資料夾」功能？
**答**：
- 團隊資料夾 = 多擁有者
- 權限繼承機制
- 稽核日誌

### ❓ Q4: 如何實現跨區域同步延遲最低？
**答**：
- 各地區有獨立 Object Storage
- 使用 RAFT 共識同步元資料
- CDN 解決靜態資源

### ❓ Q5: 如何檢測檔案是否被篡改？
**答**：
- SHA-256 雜湊值
- 用戶端可用 HMAC 驗證
- 定期完整性檢查（Integrity Check Job）

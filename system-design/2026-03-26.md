## 1️⃣ 題目說明

### 面試官會怎麼問？

> 「請設計一個公司內部的監控與警示系統，需要能監控伺服器、應用程式、資料庫的健康狀態，並在異常時發送通知。」

這題考察候選人對**可觀測性 (Observability)** 的理解，以及如何設計一個能即時發現問題、幫助工程師快速定位問題的系統。

### 需要澄清什麼問題？

在開始畫圖之前，你需要問面試官以下問題：

1. **監控範圍**：要監控哪些類型的目標？（主機、容器、K8s、應用程式、網路設備）
2. **指標類別**：需要收集哪些指標？（基礎設施指標、應用效能指標、业务指標）
3. **儲存需求**：監控資料要保留多久？（7天？30天？1年？）
4. **警示管道**：異常時要通知誰？用什麼方式？（Email、Slack、PagerDuty、短信）
5. **規模考量**：監控多少台伺服器？每秒多少筆資料？

## 2️⃣ 思考框架

### Step 1: 需求釐清

先問清楚這些問題，然後在心裡有個初步的輪廓：

- **功能需求**：收集指標 → 儲存 → 可視化 → 警示
- **非功能需求**：高可用、可擴展、低延遲
- **規模假設**：假設監控 1000 台伺服器，每台每 10 秒上報 100 個指標

### Step 3: 資料庫設計

監控系統的資料主要有三種：

#### 3.1 時序資料 (Time Series Data) - 存放指標

```sql
-- Prometheus 風格的時序資料表
CREATE TABLE metrics (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    metric_name VARCHAR(255),      -- 如: cpu_usage, memory_free
    labels JSON,                  -- 標籤: {"host": "server-01", "env": "prod"}
    value DOUBLE,                 -- 數值
    timestamp DATETIME(3),        -- 時間戳
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 優化查詢的索引
CREATE INDEX idx_metric_time ON metrics(metric_name, timestamp);
CREATE INDEX idx_labels ON metrics((JSON_EXTRACT(labels, '$.host')));
```

**比喻**：時序資料庫就像是一個**巨大的溫度記錄表**，每秒記錄每台機器的溫度，時間久了資料量會爆炸，所以需要特殊的壓縮技術。

#### 3.2 警示規則 (Alert Rules)

```sql
CREATE TABLE alert_rules (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(255),                    -- 規則名稱
    condition VARCHAR(500),               -- 條件: cpu_usage > 80
    duration INT,                         -- 持續時間（秒）
    severity ENUM('critical', 'warning', 'info'),
    description TEXT,
    enabled BOOLEAN DEFAULT TRUE,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 警示記錄
CREATE TABLE alert_events (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    rule_id INT,
    status ENUM('firing', 'resolved'),
    message TEXT,
    started_at DATETIME,
    resolved_at DATETIME,
    FOREIGN KEY (rule_id) REFERENCES alert_rules(id)
);
```

## 3️⃣ 詳細解答

### 3.1 核心元件詳解

讓我們詳細看看每個元件的角色：

#### Agent（代理程式）- 資料收集員

```python
# 簡化的 Agent  Pseudocode
class MetricAgent:
    def __init__(self):
        self.collectors = [CPUCollector(), MemoryCollector(), DiskCollector()]
        self.report_interval = 10  # 秒
    
    def collect_and_report(self):
        while True:
            metrics = []
            for collector in self.collectors:
                metrics.extend(collector.collect())
            
            # 發送到收集服務
            http.post("http://collector-service/api/v1/metrics", metrics)
            
            time.sleep(self.report_interval)
```

**比喻**：Agent 就像是公司的**保全人員**，定時巡邏並記錄每個房間的溫度、濕度、火災警報器狀態，然後回報給監控中心。

#### 時序資料庫 - 大腦記憶

常見的時序資料庫：
- **Prometheus**：Pull 模式，適合 K8s 生態
- **InfluxDB**：支援豐富的處理函數
- **TimescaleDB**：基於 PostgreSQL，易於整合
- **VictoriaMetrics**：高效能，支援長期儲存

**比喻**：時序資料庫就像是一個**超強記憶力的人**，能記住過去一年每秒的溫度變化，但不需要記得是誰測量的（這是標籤的工作）。

#### 警示引擎 - 警鈴

```python
# 警示判斷 Pseudocode
class AlertEngine:
    def evaluate(self, metric_data, rules):
        for rule in rules:
            result = self.eval_condition(metric_data, rule.condition)
            
            if result and not rule.is_firing:
                # 觸發新警示
                self.trigger_alert(rule, metric_data)
            elif not result and rule.is_firing:
                # 解除警示
                self.resolve_alert(rule)
    
    def eval_condition(self, data, condition):
        # 支援: cpu > 80, memory_usage < 20, error_rate > 0.05
        # 使用 DSL 解析條件
        return eval(condition, {"data": data})
```

### 3.2 警示進階功能

#### 警示分組 (Alert Grouping)

```yaml
# 範例：一個服務有 100 台伺服器同時掛掉
# 不希望收到 100 個警示 → 應該分組成 1 個

alert_group:
  name: "Database Cluster Down"
  group_by: ["service", "env"]  # 按服務和環境分組
  group_wait: 30s              # 等待 30 秒收集同組警示
  group_interval: 5m          # 隔 5 分鐘再發送
```

#### 警示抑制 (Alert Silencing)

```yaml
# 範例：維護期間暫停警示
silence:
  matchers:
    - name: "service"
      value: "payment"
  starts_at: "2026-03-26T10:00:00Z"
  ends_at: "2026-03-26T12:00:00Z"
  created_by: "admin@example.com"
  comment: "Payment service maintenance window"
```

### 3.3 指標類型詳解

監控指標有四大金剛：

| 類型 | 範例 | 用途 |
|------|------|------|
| **Counter** | 請求次數、錯誤次數 | 只能遞增，用於計算速率 |
| **Gauge** | CPU使用率、記憶體使用量 | 可增可減，即時值 |
| **Histogram** | 請求延遲分佈 | 統計分位数 (p50, p95, p99) |
| **Summary** | 平均延遲 | 類似 Histogram，但預先計算 |

**比喻**：
- **Counter** 像是計數器，只能一直往上加
- **Gauge** 像是溫度計，可高可低
- **Histogram** 像是把所有考生的成績分數分布圖畫出來，看分布
- **Summary** 像是只記錄平均分數，較省空間但失去分布資訊

## 4️⃣ 延伸問題

面試官可能會追問：

### Q1: 如何設計高可用的監控系統？

> 監控系統本身也會掛，如何監控監控系統？

```yaml
解決方案：
1. 資料收集：Agent 双上报（Primary + Backup）
2. 儲存層：多副本叢集（Prometheus HA Pair）
3. 警示系統：Alert Manager 叢集
4. 監控自己：讓監控系統監控自己，形成遞迴
```

### Q2: 如何處理大量指標？

> 如果有 10000 台伺服器，每台 100 個指標 = 100 萬筆/秒

```yaml
解決方案：
1. 資料彙總：在 Agent 端做預聚合
2. 採樣：高峰期降采样 (downsampling)
3. 分離熱數據和冷數據
4. 使用高效壓縮：Gorilla 壓縮可減少 90% 空間
```

### Q3: 如何確保警示不會疲勞？

> 工程師被alert炸到麻木，最後直接忽略

```yaml
解決方案：
1. 設定合理的閾值（不要隨便設 90%）
2. 使用趨勢判斷（持續 5 分鐘才報警）
3. 實施分層警示：Warning → Critical → Emergency
4. 警示分組避免重複
5. 允許設定靜音時段
```

### Q4: 如何快速定位問題？

> 收到一個 CPU 高的警示，如何快速找到原因？

```yaml
解決方案：
1. 關聯追蹤：Trace ID 串聯所有服務
2. 日誌聚合：同一時間範圍的日誌一起顯示
3. 拓撲圖：顯示服務依賴關係
4.異常指標對比：與歷史同期比較
```

### Step 3: 核心演算法 — 如何把長 URL 變短？

#### 方法 A：Hash + Base62 編碼（最常用）

```
原始想法：
  URL = "https://www.example.com/very/long/path?q=12345"
  
  1. 用 MD5/SHA256 產生 hash → 取前 8 個字元
  2. 將 8 個 16 進位字元 → 62 進位（a-z, A-Z, 0-9）
  3. 結果：6 個字元的短碼

缺點：hash 可能碰撞，需要查 DB 確認
```

#### 方法 B：Auto Increment ID（推薦 ✅）

```
核心概念：用資料庫的自動遞增 ID 來產生短碼

  ID = 1123456789
  轉成 62 進位：
    1123456789 ÷ 62 = 18120270 remainder 29  → 'd'
    18120270 ÷ 62 = 292262 remainder 26     → 'b'
    292262 ÷ 62 = 4713 remainder 56         → 'V'
    4713 ÷ 62 = 75 remainder 63             → '?'
    75 ÷ 62 = 1 remainder 13                → 'F'
    1 ÷ 62 = 0 remainder 1                  → '1'
    反轉 → "1FbVd"

✅ 優點：
  - 不會碰撞（ID 唯一）
  - 可反推（解碼容易）
  - 長度可控（62^6 = 568 億種組合）
```

### Step 4: 資料庫設計

```sql
-- 主表：短網址映射表
CREATE TABLE urls (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,  -- 流水號
    short_key VARCHAR(10) NOT NULL UNIQUE, -- 短碼 "1FbVd"
    long_url TEXT NOT NULL,                 -- 原始 URL
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    expires_at TIMESTAMP NULL,              -- 過期時間（可選）
    click_count BIGINT DEFAULT 0,           -- 點擊數
    INDEX idx_short_key (short_key)        -- 加速查詢
);

-- 可選：點擊日誌表（用於分析）
CREATE TABLE click_logs (
    id BIGINT AUTO_INCREMENT,
    short_key VARCHAR(10),
    clicked_at TIMESTAMP,
    user_ip VARCHAR(50),
    user_agent VARCHAR(255),
    INDEX idx_short_key_time (short_key, clicked_at)
);
```

### Step 6: 擴展性考量

#### 🧠 比喻理解：如何應對 100 億次/天的讀取？

```
情境：就像一個超級繁忙的圖書館
  - 很多人要借同一本書（熱門 URL）
  - 但書籍總量很大（所有 URL）

解決方案：
  1. 【Redis 熱 cache】：把最常被訪問的 URL 放記憶體
     → 圖書館前放一個暢銷書區
     
  2.【讀寫分離】：用 MySQL replica 做多個讀取節點
     → 分流查詢壓力
     
  3.【CDN】：把短網址服務放到全球 CDN
     → 世界各地都有分館
```

## 3️⃣ 詳細解答：完整流程

### 完整流程走一遍

```
1. 用戶提交：https://example.com/very/long/path

2. Server 收到請求，產生 ID = 1123456789

3. ID 轉 62 進位 → short_key = "1FbVd"

4. 寫入 DB：
   INSERT INTO urls (short_key, long_url) VALUES ('1FbVd', 'https://...')

5. 寫入 Cache：
   SET cache:1FbVd → 'https://example.com/...'

6. 返回給用戶：
   short_url = "https://tinyurl.com/1FbVd"

===

用戶訪問 https://tinyurl.com/1FbVd 時：

1. DNS 解析 → 到最近的 CDN 節點

2. CDN 檢查 Cache 有沒有？
   - 有：直接 302 重定向 ✅
   - 沒有：回源到 API Server

3. API Server 查 Redis：
   - 有：直接返回 ✅
   - 沒有：查 MySQL，返回並寫入 Cache

4. 返回 HTTP 302 + Location header
```

### 程式碼範例（Python Flask）

```python
from flask import Flask, redirect, request, jsonify
import redis
import MySQLdb

app = Flask(__name__)
r = redis.Redis(host='localhost', port=6379, db=0)

BASE62 = "0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ"

def id_to_base62(n):
    """將 ID 轉成 62 進位字串"""
    if n == 0:
        return BASE62[0]
    result = []
    while n:
        n, rem = divmod(n, 62)
        result.append(BASE62[rem])
    return ''.join(reversed(result))

@app.route('/api/shorten', methods=['POST'])
def create_short_url():
    long_url = request.json.get('long_url')
    
    # 1. 取下一個 ID（可用 Snowflake / UUID）
    next_id = get_next_id()  # 你的 ID 生成邏輯
    
    # 2. 轉成短碼
    short_key = id_to_base62(next_id)
    
    # 3. 寫入 DB
    db.execute("INSERT INTO urls (short_key, long_url) VALUES (%s, %s)", 
               (short_key, long_url))
    
    # 4. 寫入 Cache
    r.setex(f"url:{short_key}", 86400, long_url)
    
    return jsonify({
        "short_url": f"https://tinyurl.com/{short_key}",
        "short_key": short_key
    })

@app.route('/<short_key>')
def redirect_url(short_key):
    # 1. 先查 Cache
    long_url = r.get(f"url:{short_key}")
    
    if not long_url:
        # 2. Cache miss，查 DB
        result = db.execute("SELECT long_url FROM urls WHERE short_key = %s", short_key)
        if result:
            long_url = result[0]['long_url']
            # 寫回 Cache
            r.setex(f"url:{short_key}", 86400, long_url)
        else:
            return "URL not found", 404
    
    # 3. 重定向（HTTP 302）
    return redirect(long_url, code=302)
```

## 4️⃣ 延伸問題（面試官最愛追問！）

### ❓ Q1: 如何防止別人惡意生成大量短網址？
```
→ Rate Limiter（每 IP 每分鐘限制 N 次）
→ 需要登入才能使用
→ CAPTCHA 驗證
```

### ❓ Q2: 如果 ID 洩漏會怎樣？安全性怎麼設計？
```
→ 使用 Hash + 隨機鹽（不直接暴露 ID）
→ 短碼長度增加（A-Za-z0-9 共62字元，8字元 = 218兆種可能）
```

### ❓ Q3: 資料庫單點瓶頸怎麼處理？
```
→ 分片（Sharding）：按 short_key 的首字元分到不同 DB
→ 一致性雜湊（Consistent Hashing）分配
```

### ❓ Q4: 如何支援 URL 過期與刪除？
```
→ 定期清理 job（刪除 expired 的 URL）
→ 軟刪除（is_deleted flag）vs 硬刪除
→ GC 時更新 cache
```

### ❓ Q5: 如果 MySQL 掛了怎麼辦？
```
→ 主從複製（Master-Slave Replication）
→ Failover 自動切換
→ Prometheus 監控 + Alert
```

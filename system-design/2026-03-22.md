# 📚 【今日學習：設計 Twitter/X（發文、按讚、追蹤系統）】

## 1️⃣ 題目說明

### 面試官會怎麼問？

「請設計一個像 Twitter/X 的社群媒體系統。用戶可以發布推文、對推文按讚、追蹤其他用戶，並在自己的動態牆上看到追蹤者的推文。」

### 需要澄清什麼問題？

動手畫圖之前，先問清楚：

1. **流量規模**：
   - 有多少用戶？日活躍用戶（DAU）是多少？
   - 每天產生多少推文？多少按讚？

2. **功能範圍**：
   - 只需要基本功能（發文、按讚、追蹤）？
   - 還是需要轉推、回覆、hashtag、提及（@）？

3. **News Feed（動態牆）**：
   - 是按時間排序？還是演算法排序？
   - 需要「最新消息」還是「最熱門」？

4. **服務品質**：
   - 延遲要求？（看到推文需要多快）
   - 可用性？（系統不能壞）

### Step 3: 資料庫設計

Twitter 有**三張核心表**：

```sql
-- 1. 用戶表
CREATE TABLE users (
    user_id      BIGINT PRIMARY KEY,
    username     VARCHAR(50) UNIQUE,
    email        VARCHAR(100),
    created_at   TIMESTAMP
);

-- 2. 推文表
CREATE TABLE tweets (
    tweet_id     BIGINT PRIMARY KEY,
    user_id      BIGINT,           -- 誰發的
    content      VARCHAR(280),      -- 推文內容
    created_at   TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(user_id)
);

-- 3. 追蹤關係表
CREATE TABLE follows (
    follower_id  BIGINT,           -- 誰在追蹤
    followee_id  BIGINT,           -- 被追蹤的人
    created_at   TIMESTAMP,
    PRIMARY KEY (follower_id, followee_id)
);

-- 4. 按讚表
CREATE TABLE likes (
    user_id      BIGINT,
    tweet_id     BIGINT,
    created_at   TIMESTAMP,
    PRIMARY KEY (user_id, tweet_id)
);
```

**為什麼分這麼多表？**

- 想像圖書館：Users 是「作者名單」，Tweets 是「書本」，Follows 是「作者→讀者」的訂閱關係，Likes 是「讀者→書本」的收藏標記。

## 3️⃣ 詳細解答

### 🔑 核心挑戰 1：News Feed 怎麼設計？

這是 Twitter 最難的部分！當你追蹤了 500 個人，他們每天發 1000 條推文請問你要怎麼快速組合成「你的動態牆」？

**兩種主流做法：**

#### 做法 A：拉模式（Pull / Fan-out on Read）

```
用戶請求動態牆
       │
       ▼
去所有追蹤者的推文，分別抓取最新
       │
       │ 假設追蹤 500 人
       │ 就要查 500 次資料庫！
       ▼
合併、排序、回傳
```

**比喻**：像你去 500 家不同的餐廳外帶食物，然後自己組合成一桌菜。

**優點**：資料最新
**缺點**：慢！要等 500 次查詢

#### 做法 B：推模式（Push / Fan-out on Write）

```
有人發推文
       │
       │ 自動推送給所有粉絲
       ▼
寫入粉絲的「個人資料庫」
       │
       │ 粉絲讀自己的資料庫就很快
       ▼
用戶請求動態牆 → 直接讀取
```

**比喻**：像 Netflix 訂閱，你不用去找新電影，Netflix 會把新電影「推」給你。

**優點**：快！讀取只要一次
**缺點**：發文時要通知所有粉絲（計算量大）

#### 實際Twitter怎麼做？混合策略！

```python
def get_news_feed(user_id):
    """
    Twitter 的混合策略：
    - 對於粉絲少的用戶：推模式（Push）
    - 對於粉絲多的用戶：拉模式（Pull）
    """
    
    # 1. 先取得粉絲少的追蹤名單（推模式）
    following_small = get_following_with_few_followers(user_id)
    push_tweets = fetch_pushed_tweets(following_small)
    
    # 2. 再取得粉絲多的名人（拉模式）
    following_big = get_following_with_many_followers(user_id)
    pull_tweets = fetch_tweets_from_celebrities(following_big)
    
    # 3. 合併並排序
    all_tweets = push_tweets + pull_tweets
    sorted_tweets = sort_by_time(all_tweets)
    
    return sorted_tweets[:20]  # 回傳前 20 條
```

### 🔑 核心挑戰 2：如何處理高並發按讚？

想像一條病毒式推文（例如名人宣布結婚），每秒有 100 萬人按讚，資料庫會爆炸！

**解決方案：**

```
┌─────────────────────────────────────────────────────────────────┐
│                     按讚系統架構                                │
│                                                                 │
│   用戶按讚                                                        │
│      │                                                           │
│      ▼                                                           │
│   API Server                                                     │
│      │                                                           │
│      ▼                                                           │
│   Message Queue (Kafka) ───▶ 非同步寫入                        │
│      │                                                           │
│      │  批次寫入（例如每秒 1000 筆）                             │
│      ▼                                                           │
│   Counter Service (Redis) ◀─── 計數器                          │
│      │                                                           │
│      │  顯示：1,234,567                                         │
│      ▼                                                           │
│   最終寫入資料庫（慢慢同步）                                     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**關鍵概念**：最終一致性（Eventually Consistent）

- 用戶按讚 → 立刻顯示成功（樂觀回饋）
- 實際寫入資料庫 → 背景慢慢處理
- 可能延遲 1-2 秒，但用戶無感知

### 🔑 核心挑戰 3：追蹤關係怎麼儲存？

```python
# 追蹤關係的最佳化
class FollowService:
    
    def follow(self, follower_id, followee_id):
        """追蹤：用 SQL 的 JOIN 來看共同粉絲"""
        # 寫入 follows 表
        db.execute("INSERT INTO follows VALUES (?, ?)", 
                   follower_id, followee_id)
        
        # 增加雙方的計數（用 Redis 加速）
        redis.incr(f"user:{follower_id}:following")
        redis.incr(f"user:{followee_id}:followers")
    
    def get_followers(self, user_id):
        """取得粉絲名單"""
        # 從資料庫撈
        return db.query("SELECT follower_id FROM follows WHERE followee_id = ?", 
                        user_id)
    
    def suggest_follow(self, user_id):
        """推薦追蹤：用共同好友來推薦"""
        # "你的朋友也追蹤了..."
        mutual_friends = db.query("""
            SELECT f2.followee_id 
            FROM follows f1
            JOIN follows f2 ON f1.follower_id = f2.follower_id
            WHERE f1.followee_id = ?
            AND f2.followee_id != ?
            AND f2.followee_id NOT IN (SELECT followee_id FROM follows WHERE follower_id = ?)
            LIMIT 10
        """, user_id, user_id, user_id)
        
        return mutual_friends
```

### 完整程式碼範例

```python
# 簡化版 Twitter 實作
from flask import Flask, request, jsonify
import redis
import json
from datetime import datetime

app = Flask(__name__)
redis_client = redis.Redis(host='localhost', port=6379, db=0)

# 模擬資料庫（用字典）
tweets_db = {}
likes_db = {}  # tweet_id -> set of user_ids
follows_db = {}  # user_id -> set of followee_ids
tweet_counter = 0

@app.route('/api/tweets', methods=['POST'])
def create_tweet():
    """發布推文"""
    global tweet_counter
    user_id = request.json.get('user_id')
    content = request.json.get('content')
    
    if len(content) > 280:
        return jsonify({"error": "推文太長"}), 400
    
    tweet_counter += 1
    tweet_id = tweet_counter
    
    tweets_db[tweet_id] = {
        "tweet_id": tweet_id,
        "user_id": user_id,
        "content": content,
        "created_at": datetime.now().isoformat(),
        "likes": 0
    }
    
    # 將推文加入粉絲的 feed（推模式）
    if user_id in follows_db:
        for follower_id in follows_db[user_id]:
            redis_client.lpush(f"feed:{follower_id}", tweet_id)
    
    return jsonify({"tweet_id": tweet_id}), 201

@app.route('/api/tweets/<int:tweet_id>/like', methods=['POST'])
def like_tweet(tweet_id):
    """按讚"""
    user_id = request.json.get('user_id')
    
    if tweet_id not in likes_db:
        likes_db[tweet_id] = set()
    
    likes_db[tweet_id].add(user_id)
    
    # 增加計數（用 Redis 加速）
    redis_client.incr(f"tweet:{tweet_id}:likes")
    
    return jsonify({"status": "liked"})

@app.route('/api/users/<int:user_id>/follow', methods=['POST'])
def follow_user(user_id):
    """追蹤用戶"""
    target_id = request.json.get('target_id')
    
    if user_id not in follows_db:
        follows_db[user_id] = set()
    
    follows_db[user_id].add(target_id)
    
    return jsonify({"status": "followed"})

@app.route('/api/feed/<int:user_id>', methods=['GET'])
def get_feed(user_id):
    """取得動態牆"""
    # 從 Redis 拉取推文 ID
    tweet_ids = redis_client.lrange(f"feed:{user_id}", 0, 19)
    
    feed = []
    for tid in tweet_ids:
        tid = int(tid)
        if tid in tweets_db:
            tweet = tweets_db[tid].copy()
            # 取得按讚數
            tweet['likes'] = int(redis_client.get(f"tweet:{tid}:likes") or 0)
            feed.append(tweet)
    
    return jsonify({"feed": feed})

if __name__ == '__main__':
    app.run(port=8080)
```

## 4️⃣ 延伸問題

面試官可能會追問：

### Q1: 如何處理名人推文的高流量？

> **提示**：名人粉絲多，不能用「推模式」，否則要寫入 millions 筆資料。應該用「拉模式」+ 快取（Pull + Cache）。

### Q2: 如何防止刷推文/機器人？

> **提示**：Rate Limiting + 帳號年齡檢查 + 行為分析（例如每秒發超過 10 篇就阻擋）。

### Q3: 如果系統要支援「回覆」功能？

> **提示**：推文表需要增加 `parent_id` 欄位，形成樹狀結構。

### Q4: 如何設計搜尋功能？

> **提示**：不能用 SQL LIKE，需要 Elasticsearch 或 Solr 這類全文搜尋引擎。

### Q5: 如何做「已讀」功能？

> **提示**：複雜！需要追蹤每個用戶看過哪些推文，通常用 Bloom Filter 或 Redis Bitmaps。

| 關鍵概念 | 記住要點 |
|----------|----------|
| **News Feed** | 推模式（快但發文慢）vs 拉模式（慢但發文快）→ 混合策略 |

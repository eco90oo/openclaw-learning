# 📚 【今日學習：設計 Facebook News Feed（動態牆）】

## 1️⃣ 題目說明

### 面試官會怎麼問？

「請設計 Facebook 或 Instagram 的動態牆（News Feed）系統。當用戶登入後，他會看到來自朋友、粉絲專頁、按讚社團的最新貼文，並按照特定順序排列。請說明你會如何設計這個系統，讓數億用戶能夠流暢地瀏覽個人化的動態內容。」

### 需要澄清什麼問題？

在動手畫圖之前，先問清楚這些：

1. **流量規模**：
   - 有多少用戶？日活躍用戶（DAU）是多少？
   - 每個用戶平均有多少朋友/追蹤者？
   - 每天產生多少貼文？

2. **功能範圍**：
   - 動態牆是按時間排序？還是演算法排序？
   - 是否需要支援「最新動態」、「最熱門」、「追蹤中」等不同分頁？
   - 是否需要廣告摻雜在內容中？

3. **個人化程度**：
   - 需要多精準的個人化推薦？
   - 是否需要考量互動歷史（按讚、留言、分享）？

4. **服務品質**：
   - 延遲要求？（看到動態需要多快）
   - 是否需要支援「無限捲動」載入更多內容？

## 3️⃣ 詳細解答

### 3.1 核心概念：用「餐廳出菜」比喻動態牆

想象一家餐廳：
- **廚師** = 發布貼文的用戶
- **菜單** = 所有貼文的集合
- **服務生** = Feed Service
- **你的餐桌** = 你的個人動態牆

問題來了：廚師做好一道菜，不可能讓服務生每次都跑去問「所有廚師今天做了什麼菜」——那太慢了！

所以有兩種出菜方式：

### 方式 A：Pull Model（拉取）- 服務生每次去廚房問

```
用戶打開 App
    │
    ▼
請求動態牆
    │
    ▼
去所有朋友那邊「拉取」最新貼文
    │
    ▼
合併、排序、回傳
```

**優點**：資料永遠最新
**缺點**：高峰期會很慢（要等很久才能湊齊所有人的貼文）

### 方式 B：Push Model（推送）- 廚師做好菜就送到你桌上

```
用戶發布貼文
    │
    ▼
異步推送到所有粉絲的「待讀清單」
    │
    ▼
用戶打開 App → 直接從待讀清單讀取
```

**優點**：超快！打開就有
**缺點**：名人發文會瞬間推太多（fan-out explosion）

### 實際做法：混合兩種

```
一般用戶：Push（朋友少，推送很快）
名人/粉絲頁：Pull（粉絲太多，改用拉取）
```

### 3.2 資料庫設計

你需要兩套資料庫，因為「寫」和「讀」的優化方向不同：

#### 寫入 DB（User/Post 服務）

```sql
-- 用戶表
CREATE TABLE users (
    id BIGINT PRIMARY KEY,
    username VARCHAR(50),
    created_at TIMESTAMP
);

-- 貼文表
CREATE TABLE posts (
    id BIGINT PRIMARY KEY,
    user_id BIGINT,
    content TEXT,
    created_at TIMESTAMP,
    -- 索引加速查詢
    INDEX idx_user_created (user_id, created_at DESC)
);

-- 追蹤關係表
CREATE TABLE follows (
    follower_id BIGINT,
    followee_id BIGINT,
    created_at TIMESTAMP,
    PRIMARY KEY (follower_id, followee_id)
);
```

#### 讀取 DB（專門優化讀取）

```sql
-- 動態牆預計算表（每個用戶的個人化清單）
CREATE TABLE user_feed (
    user_id BIGINT,
    post_id BIGINT,
    created_at TIMESTAMP,
    score FLOAT,  -- 演算法分數
    PRIMARY KEY (user_id, created_at, post_id)
) WITH CLUSTERING ORDER BY (created_at DESC);
```

### 3.4 快取策略（Cache is King！）

動態牆是「讀多寫少」的經典場景，Cache 是關鍵：

```
┌─────────────────────────────────────────────────────┐
│              快取層級架構                            │
│                                                     │
│   L1: CDN（靜態資源：圖片、影片）                    │
│        │                                           │
│        ▼                                           │
│   L2: Redis Cluster（熱門動態牆）                   │
│        │                                           │
│        ▼                                           │
│   L3: 本地緩存（熱門貼文）                          │
│        │                                           │
│        ▼                                           │
│   DB: Cassandra（最終數據）                         │
└─────────────────────────────────────────────────────┘
```

**快取策略**：
- 每個用戶的 Feed 結果快取 5-10 分鐘
- 熱門貼文預先載入到 CDN
- 使用 Redis Sorted Set 存用戶的 Feed（按時間/分數排序）

### 3.5 排序演算法（如何決定誰先出現？）

簡單版：按時間排序
```
貼文B (12:00) → 貼文A (11:00) → 貼文C (10:00)
```

進階版：互動分數排序
```
score = like_count * 1 + comment_count * 2 + share_count * 3
      + time_decay_factor
```

真實版：機器學習模型
```
feed_score = f(用戶興趣, 貼文品質, 發文時間, 互動可能性)
```

### 3.6 處理名人效應（Fan-out Explosion）

當有一百萬粉絲的名人發文：
- Push 方式會瞬間產生 100 萬則訊息 → 系統爆炸

解決方案：

```python
def publish_post(post):
    author = get_author(post.user_id)
    
    # 粉絲少於 10 萬 → 用 Push
    if author.follower_count < 100_000:
        fanout_to_followers(post, strategy="push")
    
    # 粉絲多 → 改用 Pull
    else:
        mark_post_as_hot(post.id)
        # 用戶載入時再拉取
        
def fanout_to_followers(post, strategy):
    if strategy == "push":
        # 異步推送到每個粉絲的 feed cache
        for follower_id in get_followers(post.user_id):
            redis.zadd(f"feed:{follower_id}", post.id, score)
    # ...
```

## 4️⃣ 延伸問題

面試官可能會追問：

### Q1: 如果用戶有 5000 個朋友，該怎麼辦？
- 分頁載入，不要一次拉取所有
- 考慮取「最近的互動好友」的貼文
- 使用興趣分數篩選

### Q2: 如何處理「按讚」即時更新？
- 按讚不重新整理整個 Feed
- 使用 WebSocket 推送「XXX 對你的貼文按讚」通知
- Feed 裡的讚數可以延遲更新（最終一致性）

### Q3: 如何防止貼文順序跳來跳去？
- 分頁時使用「遊標（cursor）」而非「offset」
- 例如：last_post_id=12345&limit=20

### Q4: 如何做 A/B 測試不同的排序演算法？
- 使用 Feature Flag
- 對 5% 用戶使用新演算法
- 追蹤「停留時間」、「互動率」等指標

## 1️⃣ 基礎概念：什麼是 MVP？

### 用「賣雞蛋糕」來比喻

想像你要賣雞蛋糕：

**傳統做法（錯誤示範）**：
```
1. 研發 10 種口味（奶油、巧克力、抹茶...）
2. 設計精美包裝
3. 開實體店面
4. 請員工
5. 結果：沒人買！損失慘重
```

**MVP 做法（正確示範）**：
```
1. 先用最基本的配方做一種口味
2. 推著小攤車到處賣賣看
3. 有人買 → 研究他們喜歡什麼口味
4. 有人不買 → 問他們為什麼不買
5. 根據反饋改進
6. 確認有人要吃，再投資更多
```

**MVP = Minimum Viable Product（最小可行性產品）**

關鍵詞是「最小」+「可行」：
- **最小**：只做最核心的功能，少到不能再少
- **可行**：功能雖少，但用戶真的能用它解決問題

## 2️⃣ 為什麼 AI 時代可以 7 天驗證？

### 傳統 vs AI 時代的 MVP 開發速度

| 面向 | 傳統時代 | AI 時代 |
|------|---------|---------|
| **做一個 App** | 2-3 個月 | 1-2 週 |
| **做一個網站** | 1-2 個月 | 1-3 天 |
| **做一個原型** | 2-4 週 | 數小時 |
| **修改功能** | 1-2 週 | 數小時-1 天 |
| **獲取用戶反饋** | 1 個月 | 1 週 |

### AI 帶來的三個加速器

#### 加速器 1：程式碼生成（Code Generation）

以前需要工程師花幾天寫的功能，現在 AI 幾分鐘就能生成：

```
傳統：工程師寫程式碼 → 測試 → 修改 → 完成（3天）
AI ：告訴 AI 需求 → AI 生成 → 調整 → 完成（3小時）
```

#### 加速器 2：現成工具（No-Code / Low-Code）

AI 時代有大量現成工具，不需要從頭開發：

| 工具類型 | 例子 | 用途 |
|---------|------|------|
| 網站建置 | Carrd, Framer, Webflow | 快速 landing page |
| 表單收集 | Typeform, Google Forms | 收集用戶需求 |
| 通訊工具 | Telegram Bot, Discord Bot | 快速做出 chatbot |
| AI API | OpenAI, Anthropic, Google AI | 直接接入 AI 能力 |

#### 加速器 3：快速部署（Fast Deployment）

以前部署一個產品需要：
- 買伺服器
- 設定雲端
- 配置網域
- 設定 CI/CD

現在：
- Vercel / Netlify → 推上去就上線
- Railway / Render → 幾分鐘開好
- Streamlit / Gradio → 直接分享連結

## 3️⃣ 7 天驗證計劃：每天要做什麼？

### Day 1：定義問題與用戶

**任務**：
1. 寫下你要解決的問題（一句话）
2. 確定誰是目標用戶（具體一點）
3. 畫出最基礎的用戶流程

```
格式：

問題：[一句話描述要解決什麼問題]
目標用戶：[誰會用？年齡？職業？]
用戶流程：
1. 用戶打開產品
2. 用戶做 [某件事]
3. 用戶得到 [某個結果]
```

**比喻**：就像擺攤前先想清楚你要賣給誰、怎麼賣

### Day 2-3：做出可用的原型

**任務**：
1. 用 AI 生成第一版產品
2. 確保基本功能可以運作
3. 準備一個可以展示的版本

**推薦的快速原型工具**：

| 工具 | 適合做什麼 | 難度 |
|------|-----------|------|
| Streamlit | 數據儀表板、AI 工具 | ⭐ 簡單 |
| Gradio | AI 介面、demo | ⭐ 簡單 |
| V0 (Vercel) | Web App | ⭐⭐ 中等 |
| Bolt.new | 完整網站/App | ⭐⭐ 中等 |
| Cursor | 需要客製化 | ⭐⭐⭐ 較難 |

**具體操作（用 Streamlit 為例）**：

```python
# 今天的任務：做一個「AI 寫作助手」的原型

# 1. 首先安裝 streamlit
pip install streamlit

# 2. 建立 app.py
import streamlit as st
import openai

# 設定頁面
st.set_page_config(page_title="AI 寫作助手", page_icon="✍️")
st.title("✍️ AI 寫作助手")

# 輸入區
prompt = st.text_area("請輸入你想要寫的主題：", height=150)

# 按鈕
if st.button("生成文章"):
    if prompt:
        # 呼叫 AI（這裡假設你有 API key）
        response = openai.ChatCompletion.create(
            model="gpt-4",
            messages=[{"role": "user", "content": prompt}]
        )
        st.success("完成！")
        st.write(response.choices[0].message.content)
    else:
        st.warning("請輸入內容")

# 3. 執行
# streamlit run app.py
# 就會自動打開瀏覽器，一個 AI 產品就完成了！
```

**比喻**：就像先把攤車推出門，東西能賣再說

### Day 4：找到第一批測試用戶

**任務**：找到 5-10 個願意試用的人

**方法 1：直接問**
- 問身邊的朋友、同事
- 在社群發文：「找人測試新產品」

**方法 2：潛在用戶聚集地**
- 加入相關的 Discord/Telegram 群組
- 在 Reddit、PTT Mobile01 發文
- 在 LinkedIn 私訊目標用戶

**方法 3：用現成工具收集**
- Google Form：問他們的痛點
- Typeform：做一個更專業的調查
-Calendly：預約時間進行訪談

**模板：請求測試邀請**

```
👋 Hi，我是 [名字]，正在開發一個 [產品名稱]
這是一個幫助 [目標用戶] 解決 [痛點] 的工具

目前在做測試，想邀請你體驗一下
預計需要 15-20 分鐘
https://[產品連結]

作為回報，我會：
- 讓你免費使用一段時間
- 認真聽取你的意見
- 在產品上線後給你特別優惠

有興趣的話請回覆這則訊息，謝謝！🙏
```

### Day 5：收集並分析反饋

**任務**：和測試用戶聊天，收集意見

**要問的關鍵問題**：

| 問題 | 目的 |
|------|------|
| 「你覺得這個產品解決了你的問題嗎？」 | 驗證核心價值 |
| 「你有什麼使用上的困難？」 | 發現 bug/問題 |
| 「你願意為這個功能付多少錢？」 | 驗證付費意願 |
| 「如果這個產品消失了，你會想念它嗎？」 | 測量 retention |
| 「你會把這個產品推薦給朋友嗎？」（0-10 分） | NPS 分數 |

**反饋記錄模板**：

```
用戶：[匿名或暱稱]
背景：[年齡/職業/使用場景]

✅ 正向反饋：
- [用戶說好在哪裡]

❌ 負向反饋：
- [用戶說哪裡不好]

💡 改進建議：
- [用戶自己提的建議]

📊 行為數據：
- 用戶停留時間：[]
- 完成任務率：[]
- 是否願意付費：[]
```

### Day 6：根據反饋快速迭代

**任務**：基於收到的反饋，改進產品

**迭代原則**：

1. **優先處理「阻礙使用」的問題**
   - 如果用戶無法完成基本操作，優先修

2. **其次處理「用戶說痛」的問題**
   - 用戶說哪里不好用，優先改

3. **最後考慮「用戶建議」**
   - 用戶的建議是加分項，不是必須

**具體操作**：

```python
# 根據反饋修改 Streamlit 程式碼
# 例如：用戶說「沒有歷史記錄功能」

import streamlit as st
import openai

st.set_page_config(page_title="AI 寫作助手", page_icon="✍️")
st.title("✍️ AI 寫作助手")

# 加入：對話歷史（根據用戶反饋新增的功能）
if "history" not in st.session_state:
    st.session_state.history = []

prompt = st.text_area("請輸入你想要寫的主題：", height=150)

if st.button("生成文章"):
    if prompt:
        response = openai.ChatCompletion.create(
            model="gpt-4",
            messages=[{"role": "user", "content": prompt}]
        )
        result = response.choices[0].message.content
        
        # 存入歷史記錄
        st.session_state.history.append({
            "prompt": prompt,
            "result": result
        })
        
        st.success("完成！")
        st.write(result)

# 新增：顯示歷史記錄
if st.session_state.history:
    st.divider()
    with st.expander("📜 查看歷史記錄"):
        for i, item in enumerate(st.session_state.history):
            st.write(f"**{i+1}.** {item['prompt']}")
            st.write(item['result'])
            st.divider()
```

### Day 7：決定下一步

**任務**：根據驗證結果，決定要繼續還是 pivot

**判斷標準**：

| 指標 | 繼續開發的信號 | 應該停止/轉向的信號 |
|------|---------------|-------------------|
| **用戶意願** | >50% 用戶說「想要這個」 | 大多數人說「不需要」 |
| **使用率** | >30% 用戶每天使用 | 用戶用一次就走了 |
| **反饋品質** | 用戶給出具體建議 | 用戶說「不知道」 |
| **付費意願** | 有用户说愿意付费 | 没人愿意付钱 |
| **NPS** | >50 | <30 |

**三種可能的結果**：

1. **大成功** ✅
   - 用戶搶著要，付費意願高
   - 繼續前進，融資/擴張

2. **需要調整** ⚠️
   - 有人要但不多，需要改方向
   - Pivot（轉向）到另一個方向

3. **失敗** ❌
   - 沒人要，市場不存在
   - 承認失敗，學到教訓，重新開始

## 5️⃣ 常見陷阱與解決方法

### 陷阱 1：做太多功能

**問題**：想做 50 個功能才上線
**解決**：限制自己最多做 3 個功能

### 陷阱 2：追求完美外觀

**問題**：花太多時間在 UI/設計
**解決**：MVP 階段功能優先，外觀能看就好

### 陷阱 3：只問「會不會用」

**問題**：問「你會用這個嗎？」→ 用户都說會
**解決**：問「你願意付多少錢？」→ 這才真實

### 陷阱 4：只找朋友測試

**問題**：朋友會給你面子說好話
**解決**：找陌生人測試，或者明確說「請給真實意見」

### 陷阱 5：不敢發布

**問題**：產品不夠完美，不敢給別人看
**解決**：記住「完成 > 完美」，先發布再說

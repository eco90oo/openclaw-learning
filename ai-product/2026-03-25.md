# 📚 【AI 時代產品開發｜Day 3：7 天快速驗證與 MVP 開發】

> AI 時代最大的改變是：**你可以用極低成本快速驗證想法**。這堂課帶你掌握 7 天做出 MVP 的方法。

---

## 1️⃣ 傳統 MVP vs AI 時代 MVP

### 傳統 MVP 的問題

| 傳統做法 | 花了多少時間 |
|---------|-------------|
| 市場調研 | 2-4 週 |
| 寫 PRD | 1-2 週 |
| UI/UX 設計 | 2-4 週 |
| 開發 | 2-6 個月 |
| 測試 | 2-4 週 |
| **總計** | **3-12 個月** |

**問題**：
- 3-12 個月後，市場可能已經變了
- 花了很多時間，結果發現沒人要
- 試錯成本太高

### AI 時代的顛覆

```
7 天可以做什麼？

Day 1：確定問題 + 解決方案
Day 2-3：用 AI 生成 prototype
Day 4-5：優化核心功能
Day 6-7：找用戶測試 + 收集回饋
```

**為什麼可行？**
- AI 讓開發速度提升 10-100 倍
- 雲端服務讓基礎設施變簡單
- 現成 API 讓功能整合變容易

## 2️⃣ 7 天 MVP 實戰流程

### Day 1：問題與解決方案（最重要！）

**不要急著寫程式！先想清楚：**

```markdown
## 問題定義表

1. **我要解決什麼問題？**
   （用一句話說清楚）

2. **誰有這個問題？**
   （越具體越好：年齡、職業、場景）

3. **這個問題有多痛？**
   - 不解決會怎樣？
   - 願意花多少錢解決？

4. **現有解決方案？**
   - 他們哪裡不好？

5. **我的解決方案？**
   - 怎麼用 AI 做得更好？

6. **怎麼證明有人要？**
   - 用什麼指標？
   - 多少用戶就算成功？
```

**常見錯誤**：
- ❌ 「我要做一個 AI 產品」
- ✅ 「我要幫大學生解決論文引用格式的問題」

### Day 2-3：用 AI 生成 Prototype

**快速開發的工具組合**：

| 工具 | 用途 |
|------|------|
| Claude Code / Cursor | 寫程式 |
| Vercel | 部署 |
| Streamlit / Gradio | 快速做 UI |
| OpenAI API / Claude API | AI 能力 |
| Supabase | 資料庫 + 認證 |

**Prompt 範例**：

```
我要用 Streamlit 做一個 [產品名稱]

功能需求：
1. [功能 1]
2. [功能 2]
3. [功能 3]

技術要求：
- 使用 Python
- 用 OpenAI API 串接 GPT-4
- 簡單的 UI 即可
- 可以直接運行的完整程式碼

請給我完整的 app.py 程式碼
```

**Day 3 要做到**：
- ✅ 產品可以運行
- ✅ 核心功能可以運作
- ✅ 可以展示給別人看

### Day 4-5：優化與調整

**優化重點**：

```
1. **功能優化**
   - 用戶說哪裡不好用？
   - 哪些功能沒人用？

2. **體驗優化**
   - 操作流程順暢嗎？
   - 載入速度快嗎？

3. **穩定性**
   - 有沒有 bug？
   - error handling 做了嗎？
```

### Day 6-7：找用戶測試

**怎麼找人測試？**

| 渠道 | 做法 |
|------|------|
| 社群媒體 | Twitter/X、LinkedIn 發文 |
| 論壇 | Reddit、PTT、Dcard |
| 身邊朋友 | 直接問 |
| 付費測試 | UserTesting、Respondent |

**收集回饋的技巧**：

```markdown
## 問用戶的問題

1. 「你是做什麼的？」（了解背景）
2. 「你平時怎麼解決這個問題？」（了解現有方案）
3. 「這個產品幫你解決問題了嗎？」（核心問題）
4. 「如果有這個產品，你願意多少錢買？」（付費意願）
5. 「什麼會阻止你使用這個產品？」（最大障礙）
```

## 3️⃣ 什麼時候該 Pivot（轉向）？

### 警示信號

```
🚩 警示信號 1：
沒人願意測試你的產品
→ 可能是問題不夠痛

🚩 警示信號 2：
用戶測試後說「還不錯」
但沒人願意註冊或留下聯繫方式
→ 可能是假的興趣

🚩 警示信號 3：
用戶給了很多建議
但都是「如果做了就會用」
→ 這叫做「禮貌性建議」

🚩 警示信號 4：
對手做了類似的功能
而且做得比你好
→ 該考慮差異化了
```

### Pivot 的時機

```
✅ 應該 Pivot：
- 驗證了問題存在
- 但解決方案不對

❌ 不該 Pivot：
- 只是因為累了
- 沒有真正去驗證
```

### Pivot 的方向

| 從 | 到 |
|----|----|
| 解決太多問題 | 專注一個問題 |
| 目標用戶太廣 | 聚焦一個族群 |
| 功能太複雜 | 精簡到核心功能 |
| 免費 | 收費試試看 |

## 4️⃣ 實際案例：7 天驗證一個 AI 產品

### 案例：AI 履歷優化工具

#### Day 1：問題定義

```
問題：求職者不知道怎麼寫好履歷
目標用戶：應屆畢業生、轉職者
現有方案：104 模板、履歷健檢服務
痛點：現有方案貴（$2000+）、要等 3-5 天
```

#### Day 2-3：用 Streamlit + GPT-4 生成 prototype

```python
# 用 50 行 code 就做出來
import streamlit as st
from openai import OpenAI

st.title("AI 履歷優化助手")
resume = st.text_area("貼上你的履歷")
job_desc = st.text_area("應徵職位")

if st.button("優化"):
    # 呼叫 GPT-4
    result = client.chat.completions.create(
        model="gpt-4",
        messages=[{"role": "user", "content": f"優化這份履歷..."}]
    )
    st.write(result)
```

#### Day 4-5：優化

- 加入多個模板選擇
- 加入ATS（履歷系統）相容性檢查
- 改善輸出格式

#### Day 6-7：測試

- 在 Dcard 找工作版發文
- 邀請 20 人測試
- 收集回饋

### 結果

| 指標 | 數據 |
|------|------|
| 測試人數 | 23 人 |
| 願意註冊 | 18 人 |
| 願意付費 | 5 人 ($299/月) |
| 結論 | 值得繼續開發 |

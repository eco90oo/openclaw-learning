# 📚 Day 4：關聯式資料庫（表格如何「當好朋友」？）

---

## 1️⃣ 基礎概念

### 為什麼需要多張表？

想象一下現實生活：

**錯誤示範：把所有資料塞進一張表**

```
學生資料表（錯誤設計）
├── 學生編號：A001
├── 學生姓名：王小明
├── 選修課程：國文、數學、英文、物理、化學  <-- 問題來了！
├── 成績：國文:85、數學:90、英文:78、物理:92、化學:88
```

問題：
1. 如果學生只選 2 門課，空白要怎麼寫？
2. 如果學生要選 10 門課，寫得下嗎？
3. 想改「數學」的名称，要改所有學生的資料？

**正確示範：分散到多張表**

```
學生表（students）
├── id: 1
├── name: 王小明

課程表（courses）
├── id: 1
├── name: 國文
├── id: 2
├── name: 數學

選課表（enrollments）
├── student_id: 1   <-- 指向學生
├── course_id: 1    <-- 指向課程
├── score: 85
```

這就是**關聯式資料庫**的核心概念！

---

### 三種關係類型

#### 1️⃣ 一對一（One-to-One）

**比喻**：一夫一妻制

```
先生表 ←→ 太太表
一位先生對一位太太
```

**範例**：
- user（用戶基本資料）↔ user_profile（用戶詳細資料）
- person（基本信息）↔ passport（護照資訊）

**實際應用**：把不常用的欄位拆分到另一張表，提高效能

#### 2️⃣ 一對多（One-to-Many）

**比喻**：一個媽媽有很多小孩

```
媽媽表 ←→ 小孩表
一個媽媽可以有很多小孩
一個小孩只有一個媽媽
```

**範例**：
- 部門（1）→ 員工（多）
- 分類（1）→ 商品（多）
- 作者（1）→ 書籍（多）

**實作關鍵**：在「多」的這邊新增欄位，存「一」的 ID

```
-- 員工表（多）需要有 department_id
employees
├── id: 1
├── name: 小李
├── department_id: 1  <-- 指向部門

-- 部門表（1）
departments
├── id: 1
├── name: 工程部
```

#### 3️⃣ 多對多（Many-to-Many）

**比喻**：學生選課

```
學生表 ←→ 選課表 ←→ 課程表
一個學生可以選很多課
一門課可以有很多學生選
```

**範例**：
- 學生 ↔ 課程
- 會員 ↔ 課程
- 作者 ↔ 書籍

**實作關鍵**：需要一張「中間表」來連接

```
-- 中間表（junction table）
enrollments
├── student_id  <-- 指向學生
├── course_id   <-- 指向課程
├── enroll_date <-- 選課日期
└── score       <-- 成績
```

### ER 圖（Entity-Relationship Diagram）

**比喻**：資料庫的設計圖

就像蓋房子前要先畫設計圖，資料庫也要先畫 ER 圖！

**符號說明**：

```
┌─────────┐       1       N       ┌─────────┐
│  部門   │───────□──────────→│  員工   │
└─────────┘                   └─────────┘
   實體（Entity）               屬性（Attribute）
```

- **實體（Entity）**：一張表（比如部門、員工）
- **屬性（Attribute）**：欄位（比如姓名、年齡）
- **關係（Relationship）**：表與表之間的連結

**常見標註方式**：

```
一對一：1 ----- 1
一對多：1 ----- N
多對多：N ----- M
```

---

### 外鍵（Foreign Key）是什麼？

**比喻**：身份證

```
員工表的身分證：
├── employee_id: 1
├── name: 小李
├── department_id: 3  <-- 這就是「外鍵」！
                     就像身份證上寫「爸爸是誰」
```

**外鍵 = 指向其他表主鍵的欄位**

作用：
1. **確保資料完整性**：不能亂寫 department_id，必須是部門表中存在的 ID
2. **建立關聯**：讓兩張表連在一起

## 2️⃣ SQL 語法實作

### 情境：我們要建立「部門-員工」系統

需求：
- 部門表（departments）：部門編號、名稱
- 員工表（employees）：員工編號、姓名、薪水、所屬部門

### 建立表格（一步步來）

```sql
-- Step 1：先建立部門表（一方）
CREATE TABLE departments (
    id INTEGER PRIMARY KEY,           -- 部門流水號，主鍵
    dept_no TEXT UNIQUE NOT NULL,     -- 部門編號（如 D001），唯一且必填
    dept_name TEXT NOT NULL           -- 部門名稱，必填
);

-- Step 2：再建立員工表（多方）
CREATE TABLE employees (
    id INTEGER PRIMARY KEY,           -- 員工流水號，主鍵
    emp_no TEXT UNIQUE NOT NULL,      -- 員工編號（如 E001），唯一且必填
    name TEXT NOT NULL,               -- 員工姓名，必填
    salary REAL,                      -- 薪水，可空白
    department_id INTEGER,            -- 所屬部門（外鍵）
    
    -- 關鍵語法：建立外鍵約束
    -- 意思是：department_id 必須參考 departments 表的 id
    FOREIGN KEY (department_id) REFERENCES departments(id)
);
```

**解讀 FOREIGN KEY 語法：**

```sql
FOREIGN KEY (department_id)           -- 在 employees 表的 department_id 欄位
REFERENCES departments(id)            -- 參考 departments 表的 id 欄位
```

翻成白話：「這個欄位只能寫 departments 表中存在的部門編號」

### 查詢關聯資料（JOIN）

**重點來了！** 如何把兩張表連在一起查？

```sql
-- 語法：SELECT * FROM 表A JOIN 表B ON 條件

-- 查詢員工和部門資訊
SELECT 
    employees.name AS 員工姓名,        -- 員工表的 name，取名為「員工姓名」
    employees.salary AS 薪水,           -- 員工表的 salary
    departments.dept_name AS 部門      -- 部門表的 dept_name
FROM employees                         -- 從 employees 表開始
JOIN departments ON                    -- 連接到 departments 表，條件是：
    employees.department_id = departments.id;
```

**結果：**

| 員工姓名 | 薪水 | 部門 |
|----------|------|------|
| 小李 | 50000 | 工程部 |
| 小王 | 45000 | 業務部 |
| 小美 | 35000 | （空白）|

**白話解釋 JOIN：**

```
想像你在翻一本相簿：
- employees 是左頁（有員工照片）
- departments 是右頁（有部門介紹）
- JOIN 就是把兩頁攤開，中間用膠水黏起來
- ON 條件就是：「找到左頁那個人所屬的部門」
```

### 多對多關係實作

**情境：學生選課系統**

需要三張表：
1. students（學生表）
2. courses（課程表）
3. enrollments（選課表，中間表）

```sql
-- 建立學生表
CREATE TABLE students (
    id INTEGER PRIMARY KEY,
    student_no TEXT UNIQUE NOT NULL,
    name TEXT NOT NULL
);

-- 建立課程表
CREATE TABLE courses (
    id INTEGER PRIMARY KEY,
    course_no TEXT UNIQUE NOT NULL,
    course_name TEXT NOT NULL,
    credits INTEGER            -- 學分數
);

-- 建立選課表（中間表，兩個外鍵）
CREATE TABLE enrollments (
    id INTEGER PRIMARY KEY,
    student_id INTEGER NOT NULL,    -- 指向學生
    course_id INTEGER NOT NULL,     -- 指向課程
    enroll_date TEXT,               -- 選課日期
    score INTEGER,                  -- 成績
    
    -- 兩個外鍵約束
    FOREIGN KEY (student_id) REFERENCES students(id),
    FOREIGN KEY (course_id) REFERENCES courses(id)
);
```

**插入測試資料：**

```sql
-- 插入學生
INSERT INTO students (student_no, name) VALUES ('S001', '王小明');
INSERT INTO students (student_no, name) VALUES ('S002', '李美華');

-- 插入課程
INSERT INTO courses (course_no, course_name, credits) VALUES ('C001', '資料庫', 3);
INSERT INTO courses (course_no, course_name, credits) VALUES ('C002', '程式設計', 4);
INSERT INTO courses (course_no, course_name, credits) VALUES ('C003', '網頁設計', 2);

-- 學生選課（中間表）
INSERT INTO enrollments (student_id, course_id, enroll_date, score)
VALUES (1, 1, '2024-09-01', 85);  -- 王小明選資料庫，85分

INSERT INTO enrollments (student_id, course_id, enroll_date, score)
VALUES (1, 2, '2024-09-01', 90);  -- 王小明選程式設計，90分

INSERT INTO enrollments (student_id, course_id, enroll_date, score)
VALUES (2, 1, '2024-09-01', 78);  -- 李美華選資料庫，78分
```

**查詢：某學生選了哪些課？**

```sql
SELECT 
    students.name AS 學生姓名,
    courses.course_name AS 課程名稱,
    courses.credits AS 學分,
    enrollments.score AS 成績
FROM enrollments
JOIN students ON enrollments.student_id = students.id
JOIN courses ON enrollments.course_id = courses.id
WHERE students.name = '王小明';
```

**結果：**

| 學生姓名 | 課程名稱 | 學分 | 成績 |
|----------|----------|------|------|
| 王小明 | 資料庫 | 3 | 85 |
| 王小明 | 程式設計 | 4 | 90 |

## 3️⃣ 常見錯誤/雷點提醒

### ⚠️ 雷點 1：先有「多」還是先有「一」？

**錯誤順序：**
```sql
-- 先建員工表（但部門還沒建，會出問題）
CREATE TABLE employees (
    id INTEGER PRIMARY KEY,
    department_id INTEGER,
    FOREIGN KEY (department_id) REFERENCES departments(id)  -- 部門表呢？
);
```

**正確順序：**
```sql
-- 先建「一」方（部門）
CREATE TABLE departments ( ... );

-- 再建「多」方（員工）
CREATE TABLE employees ( ... );
```

---

### ⚠️ 雷點 2：外鍵約束失效？

**SQLite 預設不檢查外鍵！**

```sql
-- 強制開啟外鍵檢查（SQLite 專用）
PRAGMA foreign_keys = ON;

-- 測試：故意寫一個不存在的部門 ID
INSERT INTO employees (emp_no, name, department_id)
VALUES ('E999', '幽靈員工', 999);  -- 部門 ID 999 不存在！
-- 結果：如果開了 foreign_keys，會失敗
-- 結果：如果沒開，會成功（但資料是錯的！）
```

**提醒：** SQLite 要手動開啟外鍵檢查！

---

### ⚠️ 雷點 3：刪除「一」時怎麼辦？

如果我要刪除「工程部」，員工怎麼辦？

```sql
-- 預設行為：可以刪除（員工的 department_id 變成 NULL）
-- 這叫 ON DELETE SET NULL

-- 如果要阻止刪除（部門下面還有人）：
-- 這叫 ON DELETE RESTRICT

-- 如果要跟著刪除（部門解散，員工也消失）：
-- 這叫 ON DELETE CASCADE
```

```sql
CREATE TABLE employees (
    ...
    department_id INTEGER,
    FOREIGN KEY (department_id) 
        REFERENCES departments(id) 
        ON DELETE CASCADE   -- 部門刪除，員工也刪除
);
```

### ⚠️ 雷點 4：多對多查詢結果爆炸？

如果課程很多、学生很多：
- 10 個學生 × 5 門課 = 50 筆記錄
- 查詢時可能一次顯示很多行

這不是錯誤，但要注意！

## 💡 小知識

**為什麼要用外鍵？**

不用外鍵也能查詢：

```sql
-- 不用外鍵（自己記得就好）
SELECT * FROM employees, departments 
WHERE employees.dept_id = departments.id;

-- 用外鍵（資料庫幫你檢查）
SELECT * FROM employees 
JOIN departments ON employees.dept_id = departments.id;
```

差異：
- **不用外鍵**：寫錯也沒人發現，資料可能不一致
- **用外鍵**：資料庫幫你把關，寫錯會阻止你

**記住：懶得設外鍵是以後麻煩的開始！**

# 📚 Day 5：正規化（讓資料庫不再「凌凌刁刁」）

---

## 1️⃣ 基礎概念

### 什麼是正規化？

**比喻：整理衣櫃**

想象你有一個雜亂的衣櫃：
- 襪子和內衣放在一起
- 夏天衣服塞在冬天衣服下面
- 找不到想穿的那件衣服

正規化就像是**把衣櫃分門別類**：
- 上層：夏天衣服
- 中層：冬天衣服
- 下層：內衣襪子
- 抽屜：配件

這樣要找什麼都很容易！

### 為什麼需要正規化？

**問題：資料重複造成的麻煩**

```
學生課程表（未正規化）
┌──────┬───────┬───────┬──────┬────────┐
│ 學號 │ 姓名  │ 課程  │ 學分 │ 老師   │
├──────┼───────┼───────┼──────┼────────┤
│ S001 │ 王小明│ 資料庫│ 3    │ 張老師 │
│ S001 │ 王小明│ 程式  │ 4    │ 李老師 │
│ S002 │ 李美華│ 資料庫│ 3    │ 張老師 │
│ S003 │ 張志強│ 資料庫│ 3    │ 張老師 │
└──────┴───────┴───────┴──────┴────────┘

問題來了：
1. 「資料庫」出現 3 次，學分「3」也出現 3 次
2. 如果要修改「資料庫」的學分，要改 3 個地方！
3. 「張老師」出現 3 次，如果換老師，要改 3 次
4. 新增一門課，沒有學生選，也要寫？
5. 刪除所有學生，課程資料不見了？
```

### 正規化的目標

```
🎯 目標 1：消除資料重複（每筆資料只存一次）
🎯 目標 2：確保資料一致性（修改一次就生效）
🎯 目標 3：避免更新異常（新增、刪除不會弄丟資料）
```

## 2️⃣ 正規化階段（Normal Forms）

正規化有點像打遊戲通關，一共有 6 關（通常只用到前 3 關）：

| 關卡 | 名稱 | 目標 |
|------|------|------|
| 第一關 | 1NF | 消除重複的欄位群組 |
| 第二關 | 2NF | 消除部分依賴（移除冗餘） |
| 第三關 | 3NF | 消除遞移依賴 |
| 第四關 | BCNF | 更嚴格的 3NF |
| 第五關 | 4NF | 消除多值依賴 |
| 第六關 | 5NF | 消除連接依賴 |

### 第一關：1NF（第一正規化）

**要求**：每個欄位只能存「一個值」

**錯誤示範**（一個欄位存多個值）：
```
┌──────┬───────┬───────────────┐
│ 學號 │ 姓名  │ 選修課程      │
├──────┼───────┼───────────────┤
│ S001 │ 王小明│ 資料庫,程式,網頁│  <-- 這是「多個值」
└──────┴───────┴───────────────┘
```

**正確示範**（分開存放）：
```
┌──────┬───────┬────────────┐
│ 學號 │ 姓名  │ 選修課程   │
├──────┼───────┼────────────┤
│ S001 │ 王小明│ 資料庫     │
│ S001 │ 王小明│ 程式       │
│ S001 │ 王小明│ 網頁       │
└──────┴───────┴────────────┘
```

**白話**：不要把「多個東西」塞進一個欄位！

---

### 第二關：2NF（第二正規化）

**前提**：已經通過 1NF
**要求**：消除「部分依賴」

**什麼是部分依賴？**

想象你有這樣的表：
```
選課表（學號 + 課程編號 是主鍵）
┌──────┬──────────┬──────┬────────┐
│ 學號 │ 課程編號 │ 課程名│ 學分   │
├──────┼──────────┼──────┼────────┤
│ S001 │ C001     │ 資料庫│ 3     │
│ S001 │ C002     │ 程式  │ 4     │
│ S002 │ C001     │ 資料庫│ 3     │
└──────┴──────────┴──────┴────────┘

問題：
- 「課程名稱」和「學分」只跟「課程編號」有關
- 跟「學號」沒關係！
- 這叫「部分依賴」

解決方法：拆成兩張表
┌─────────────┐     ┌─────────────┐
│ 選課表      │     │ 課程表      │
├─────────────┤     ├─────────────┤
│ 學號        │     │ 課程編號    │
│ 課程編號    │─────→│ 課程名稱    │
│             │     │ 學分        │
└─────────────┘     └─────────────┘
```

**白話**：把「只跟某個欄位有關」的資料拆出去！

---

### 第三關：3NF（第三正規化）

**前提**：已經通過 2NF
**要求**：消除「遞移依賴」

**什麼是遞移依賴？**

```
員工表
┌──────┬───────┬─────────┬────────┐
│ 員工編│ 姓名  │ 部門編號│ 部門名 │
├──────┼───────┼─────────┼────────┤
│ E001 │ 小李  │ D001    │ 工程部 │
│ E002 │ 小王  │ D002    │ 業務部 │
└──────┴───────┴─────────┴────────┘

問題分析：
- 「部門名」依賴「部門編號」
- 「部門編號」依賴「員工編號」（主鍵）
- 所以「部門名」遞移依賴「員工編號」

如果換部門名稱，要改所有有這個部門的員工！

解決方法：拆成兩張表
┌─────────────┐     ┌─────────────┐
│ 員工表      │     │ 部門表      │
├─────────────┤     ├─────────────┤
│ 員工編號    │     │ 部門編號    │
│ 姓名        │     │ 部門名      │
│ 部門編號    │─────→│            │
└─────────────┘     └─────────────┘
```

**白話**：不要讓非主鍵欄位「間接」依賴主鍵！

## 3️⃣ 實際操作

### 案例：設計「訂單系統」

**需求**：
- 訂單編號
- 客戶名稱
- 客戶電話
- 商品名稱
- 商品單價
- 購買數量
- 總金額
- 訂單日期

### Step 1：畫出未正規化的表格

```
┌────────┬───────┬───────┬───────┬──────┬──────┬──────┬────────┐
│ 訂單編號│ 客戶名│ 電話  │ 商品名│ 單價 │ 數量 │ 總金額│ 日期   │
├────────┼───────┼───────┼───────┼──────┼──────┼──────┼────────┤
│ O001   │ 王小明│ 0912  │ iPhone│ 30000│ 2    │ 60000│ 3/24   │
│ O001   │ 王小明│ 0912  │ AirPods│ 5000│ 1    │ 60000│ 3/24   │
│ O002   │ 李美華│ 0922  │ iPad  │ 20000│ 1    │ 20000│ 3/24   │
└────────┴───────┴───────┴───────┴──────┴──────┴──────┴────────┘

問題：
1. 王小明出現兩次（重複）
2. 電話 0912 重複
3. 總金額要自己算（不是從單價×數量來的）
4. 訂單日期重複
```

### Step 2：正規化到 1NF

**原則**：每個欄位只存一個值

但這個表其實已經是 1NF 了（每格一個值），只是有重複問題。

### Step 3：正規化到 2NF

**原則**：消除部分依賴（找出可獨立的資料）

分析：
- 「客戶名、客戶電話」只跟「訂單編號」有關 → 客戶表
- 「商品名、單價」只跟「商品編號」有關 → 商品表
- 「數量、總金額」跟「訂單編號 + 商品編號」都有關 → 訂單明細表

### Step 4：正規化到 3NF

**原則**：消除遞移依賴

```
訂單表（orders）
├── order_id（主鍵）
├── customer_id（外鍵）
└── order_date

客戶表（customers）
├── customer_id（主鍵）
├── customer_name
└── customer_phone

商品表（products）
├── product_id（主鍵）
├── product_name
└── price

訂單明細表（order_items）
├── order_id（外鍵，指向訂單）
├── product_id（外鍵，指向商品）
└── quantity（數量）
```

---

### 實際建立表格

```sql
-- 客戶表
CREATE TABLE customers (
    id INTEGER PRIMARY KEY,
    name TEXT NOT NULL,
    phone TEXT
);

-- 商品表
CREATE TABLE products (
    id INTEGER PRIMARY KEY,
    name TEXT NOT NULL,
    price INTEGER NOT NULL
);

-- 訂單表
CREATE TABLE orders (
    id INTEGER PRIMARY KEY,
    customer_id INTEGER NOT NULL,
    order_date TEXT NOT NULL,
    FOREIGN KEY (customer_id) REFERENCES customers(id)
);

-- 訂單明細表（關鍵的中間表！）
CREATE TABLE order_items (
    id INTEGER PRIMARY KEY,
    order_id INTEGER NOT NULL,
    product_id INTEGER NOT NULL,
    quantity INTEGER NOT NULL,
    FOREIGN KEY (order_id) REFERENCES orders(id),
    FOREIGN KEY (product_id) REFERENCES products(id)
);
```

### 插入測試資料

```sql
-- 先有客戶
INSERT INTO customers (name, phone) VALUES ('王小明', '0912345678');
INSERT INTO customers (name, phone) VALUES ('李美華', '0922345678');

-- 再有商品
INSERT INTO products (name, price) VALUES ('iPhone', 30000);
INSERT INTO products (name, price) VALUES ('AirPods', 5000);
INSERT INTO products (name, price) VALUES ('iPad', 20000);

-- 建立訂單
INSERT INTO orders (customer_id, order_date) VALUES (1, '2026-03-24');

-- 訂單裡買了什麼
INSERT INTO order_items (order_id, product_id, quantity) VALUES (1, 1, 2);
INSERT INTO order_items (order_id, product_id, quantity) VALUES (1, 2, 1);

-- 再來一筆訂單
INSERT INTO orders (customer_id, order_date) VALUES (2, '2026-03-24');
INSERT INTO order_items (order_id, product_id, quantity) VALUES (2, 3, 1);
```

### 查詢：訂單資訊

```sql
-- 查詢某筆訂單的完整資訊
SELECT 
    orders.id AS 訂單編號,
    customers.name AS 客戶名稱,
    products.name AS 商品名稱,
    products.price AS 單價,
    order_items.quantity AS 數量,
    (products.price * order_items.quantity) AS 小計,
    orders.order_date AS 日期
FROM orders
JOIN customers ON orders.customer_id = customers.id
JOIN order_items ON orders.id = order_items.order_id
JOIN products ON order_items.product_id = products.id;
```

**結果：**

| 訂單編號 | 客戶名稱 | 商品名稱 | 單價 | 數量 | 小計 | 日期 |
|----------|----------|----------|------|------|------|------|
| 1 | 王小明 | iPhone | 30000 | 2 | 60000 | 2026-03-24 |
| 1 | 王小明 | AirPods | 5000 | 1 | 5000 | 2026-03-24 |
| 2 | 李美華 | iPad | 20000 | 1 | 20000 | 2026-03-24 |

## 💡 小知識

**正規化不是越高越好！**

```
✅ 正規化優點：
- 資料不重複
- 更新方便
- 節省空間

❌ 正規化缺點：
- 查詢要 JOIN 很多表
- 有時候會變慢

⚖️ 建議：
- 一般系統：做到 3NF
- 讀取為主的系統：可以適度反正規化
- 交易系統：嚴格正規化
```

記住：**正規化是工具，不是教條！**

# 📚 【今日學習：Day 1 - 資料庫基礎（什麼是資料庫、為什麼要學）】

## 1️⃣ 基礎概念（用生活比喻，越詳細越好）

### 🏠 資料庫是什麼？用生活比喻來解釋！

想像一下，你要去圖書館找一本書。

**沒有資料庫的時代（想像你家裡的書桌亂成一團）：**
- 書本隨便堆在角落
- 要找某本書得翻遍整個房間
- 同事借了你的書忘了還，你也不記得了
- 書本可能被蟲蛀、被水淹，沒有備份

**有資料庫的時代（像圖書館有系統管理）：**
- 每本書都有編號（ISBN）
- 借書、還書都有紀錄
- 可以用關鍵字快速搜尋
- 系統會自動提醒逾期未還
- 圖書館會定期備份，防止資料遺失

**🏆 資料庫就是這麼一個「有系統的電子圖書館」！**

---

### 📊 資料庫的組成元件（讓你更清楚它在幹嘛）

| 元件 | 生活比喻 | 實際意義 |
|------|---------|---------|
| **資料庫（Database）** | 整座圖書館 | 存放資料的容器 |
| **表格（Table）** | 圖書館裡的書架 | 存放特定類型的資料 |
| **列（Row/Record）** | 書架上的一本書 | 一筆資料 |
| **欄位（Column/Field）** | 書的屬性（書名、作者、頁數） | 資料的欄位 |
| **欄位類型（Data Type）** | 這格放文字、那格放數字 | 資料的型態 |

---

### 💡 為什麼要學資料庫？（這年頭必備技能！）

#### 理由一：所有 App 都需要資料庫
- 你用手機滑的 Facebook、Instagram、Spotify
- 網路購物的露天、momo、PChome
- 甚至你家裡的智慧家庭的設定

**全部都需要資料庫來儲存用戶資料、訂單、庫存！**

#### 理由二：處理「大量資料」的唯一解
Excel 只能處理幾萬筆資料，但一個電商網站可能有幾百萬筆訂單！
資料庫專門設計來處理這種海量資料。

#### 理由三：多人同時使用也不會壞（併發控制）
- 10 個人同時搶購同一個商品，資料庫能確保每個人看到的庫存都正確
- 不會出現「超賣」的問題
- 這就是我們後面會教的「鎖機制」和「交易（Transaction）」

#### 理由四：資料安全又可靠
- 突然當機？資料庫有「交易日誌」可以復原
- 怕資料不見？資料庫可以做「備份」
- 怕被駭客看？資料庫有「權限管理」

### 🔄 同步（Synchronous）vs 非同步（Asynchronous）- 先有個概念！

這兩個概念在資料庫世界非常重要，讓我用「點餐」來比喻：

**同步（Synchronous）＝ 餐廳外帶：**
```
你點餐 -> 廚師開始做 -> 你站在那裡等 -> 做好給你 -> 你才能去做其他事
```
在資料庫裡：一次只能做一件事，做完才能做下一個。

**非同步（Asynchronous）＝ 美食外送 App：**
```
你點餐 -> 系統說「已收到訂單」-> 你可以去上班/看電影 -> 廚師做好 -> 外送員送到
```
在資料庫裡：可以同時處理很多請求，不用乾等。

**📌 重點：現代資料庫應用幾乎都是非同步的！Day 15 我們會詳細實作！**

## 2️⃣ 資料庫的種類（先有整體概念）

### 🗂️ 關聯式資料庫（Relational Database）- 最主流！

就像 Excel 的表格，但更強大！

| 資料庫 | 簡介 | 適合場景 |
|-------|------|---------|
| **MySQL** | 最流行的開源資料庫 | Web 網站、部落格 |
| **PostgreSQL** | 功能最齊全的開源資料庫 | 企業級應用、GIS |
| **Microsoft SQL Server** | 微軟出品，企業常用 | Windows 企業系統 |
| **Oracle** | 資料庫界的老大 | 銀行、證券 |

### 📄 文件資料庫（NoSQL）- 適合海量非結構化資料

| 資料庫 | 簡介 | 適合場景 |
|-------|------|---------|
| **MongoDB** | 文件資料庫，用 JSON 存資料 | 部落格、用戶行為分析 |
| **Redis** | 鍵值資料庫，超級快 | 快取、訊息佇列、即時排行榜 |

## 3️⃣ 實際操作（一步一步帶著做）

### 環境準備：我們用 SQLite 來練習！

SQLite 是最輕量的資料庫，檔案形式存在，不需要安裝伺服器，非常適合初學者！

```bash
# macOS / Linux 安裝（通常已內建）
# 檢查是否有 sqlite3
sqlite3 --version
```

### 啟動 SQLite 互動模式

```bash
sqlite3 my_first_database.db
```

你會看到：
```
SQLite version 3.xx.x
Enter ".help" for instructions.
Enter SQL statements terminated with a ";"
```
表示你已进入 SQLite 世界了！

### 建立第一個表格（就像在圖書館創建一個書架）

```sql
-- 建立一個名為 "users" 的表格（用來存放用戶資料）
-- 我們會在 Day 3 詳細解釋每一行的意義
CREATE TABLE users (
    id INTEGER PRIMARY KEY,           -- 這是用戶的編號（主鍵），像圖書館的索引號碼
    name TEXT NOT NULL,               -- 用戶姓名（文字，不能空白）
    email TEXT UNIQUE NOT NULL,       -- 用戶 email（不能重複）
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP  -- 建立時間（自動產生）
);
```

### 插入一些測試資料

```sql
-- 插入第一個用戶（就像在書架上放第一本書）
INSERT INTO users (name, email) VALUES ('小明', 'ming@example.com');

-- 插入第二個用戶
INSERT INTO users (name, email) VALUES ('小華', 'hua@example.com');

-- 插入第三個用戶
INSERT INTO users (name, email) VALUES ('小美', 'mei@example.com');
```

### 查看資料（就像在圖書館查詢書籍）

```sql
-- 查看 users 表格裡的所有資料
SELECT * FROM users;
```

你應該會看到：
```
id          name        email                   created_at
----------  ----------  ----------------------  -------------------
1           小明        ming@example.com        2026-03-28 00:00:00
2           小華        hua@example.com         2026-03-28 00:00:00
3           小美        mei@example.com         2026-03-28 00:00:00
```

### 查看表格結構

```sql
-- 看一下這個表格長什麼樣子（有哪幾個欄位）
.schema users
```


## 4️⃣ 常見錯誤/雷點提醒

### ❌ 錯誤一：把資料庫當作 Excel 用
很多人把資料庫當作 Excel 在用，結果做出來的表格混乱不堪。
**記住：資料庫是有「關係」的！Day 4 會教你怎么設計關聯！**

### ❌ 錯誤二：忘記 PRIMARY KEY（主鍵）
每個表格都要有一個主鍵，用來唯一識別每一筆記錄。
就像每本書都有 ISBN 一樣！

### ❌ 錯誤三：取了奇怪的名字
- 千萬不要用中文當欄位名（如：姓名）
- 千萬不要用空格（如：user name）
- 建議用小寫加底線（如：user_name）

### ❌ 錯誤四：SQL 語法大小寫混用
- 標準做法：關鍵字大寫（如 SELECT、INSERT）
- 表格名稱小寫（如 users、orders）
- 這樣比較好閱讀！

# 📚 Day 3：表格設計（Table、Column、Primary Key）

---

## 1️⃣ 基礎概念（用生活比喻）

### 什麼是表格設計？

**比喻：蓋房子打地基**

蓋房子前要先畫設計圖，資料庫也是一樣：
- 表格（Table）= 房子的房間
- 欄位（Column）= 房間裡的設施（門、窗、電源）
- 主鍵（Primary Key）= 房間的門牌號碼

**設計好表格 = 蓋好房子的骨架**
沒設計好，之後改代價很大！

### 表格 vs 欄位 vs 列

```
📋 表格：students（學生）

┌─────────────────────────────────────┐
│  students（學生表）                  │
├──────────┬─────────┬───────┬────────┤
│  id       │ name    │ age   │ email  │
│  (欄位)   │ (欄位)  │(欄位) │ (欄位) │
├──────────┼─────────┼───────┼────────┤
│  1        │ 小明    │  18   │ a@..   │  ← 這是一列（Row）
│  2        │ 小華    │  19   │ b@..   │  ← 這是一列（Row）
│  3        │ 小美    │  20   │ c@..   │  ← 這是一列（Row）
└──────────┴─────────┴───────┴────────┘
```

### 為什麼設計這麼重要？

```
❌ 設計不好的例子：
├── 同一個欄位，有時存數字，有時存文字（混亂）
├── 客戶地址分散在 3 個欄位（難找）
├── 沒有主鍵，資料可能重複（災難）
└── 電話欄位叫 phone，有的地方叫 tel（不一致）

✅ 設計好的例子：
├── 每個欄位型態清楚（數字就存數字）
├── 主鍵唯一識別每一筆資料
├── 欄位命名一致且有意義
└── 未來擴充容易
```

## 2️⃣ 欄位資料類型

### 常用資料類型（以 MySQL/SQLite 為例）

| 類型 | 用途 | 比喻 | 範例 |
|------|------|------|------|
| INT / INTEGER | 整數 | 碗公（裝整數） | 年齡、數量、價格 |
| VARCHAR(n) | 可變長度文字 | 彈性氣球（多少裝多少） | 名字、地址 |
| TEXT | 長文字 | 游泳池（裝很多字） | 文章、留言 |
| DECIMAL(n,m) | 精確小數 | 算盤（精準計算） | 金錢（100.50） |
| DATE | 日期 | 日曆 | 生日、訂單日期 |
| DATETIME | 日期時間 | 農民曆 | 建立時間 |
| BOOLEAN | 是/否 | 開/關開關 | 是否啟用 |
| BLOB | 二進制資料 | 倉庫（裝任何東西） | 圖片、檔案 |

### 比喻解說

```
INT 就像是量杯：
- 裝 100ml 水（100）
- 裝 500ml 水（500）
- 不能裝「半杯」或「文字」

VARCHAR 就像是橡皮筋：
- 可以綁 1 個蘋果
- 可以綁 100 個蘋果
- 但有長度上限（100個字以內）

TEXT 就像是備忘錄：
- 理論上可以寫無限多字
- 但查詢會比較慢（少用為妙）
```

## 3️⃣ 主鍵（Primary Key）

### 什麼是主鍵？

**比喻：身份證字號**

每個台灣人都有唯一的身份證字號：
- 不能重複 ✅
- 不能空白 ✅
- 用它就能找到你這個人 ✅

資料庫的主鍵也是一樣：
- **唯一** - 不能有兩筆資料有一樣的主鍵
- **不能為空** - 必須有值
- **用於識別** - 就像身份證找人一樣

### 主鍵的選擇

```sql
-- 方法 1：使用自然主鍵（有意義的欄位）
-- 例如：身份證字號、員工編號
CREATE TABLE employees (
    emp_id VARCHAR(10) PRIMARY KEY,  -- 員工編號，如 E001
    name TEXT,
    department TEXT
);

-- 方法 2：使用代理主鍵（自動編號，沒意義但簡單）
-- 大多數情況用這個！
CREATE TABLE students (
    id INTEGER PRIMARY KEY AUTOINCREMENT,  -- 自動產生 1, 2, 3...
    name TEXT,
    age INTEGER
);

-- 方法 3：複合主鍵（用兩個欄位當主鍵）
-- 很少用，但有時需要
CREATE TABLE enrollments (
    student_id INTEGER,
    course_id INTEGER,
    grade TEXT,
    PRIMARY KEY (student_id, course_id)  -- 一個學生對一門課只能選一次
);
```

### 主鍵小提醒

```
✅ 最佳實踐：
- 大部分情況用「自動編號整數」
- 不要用「可能會變」的值當主鍵（例如：Email 以後可能變）
- 主鍵一旦設錯，後面麻煩很多

❌ 常見錯誤：
- 用名字當主鍵（重名怎麼辦？）
- 用電話當主鍵（換電話怎麼辦？）
- 讓主鍵可以重複
```

## 4️⃣ 約束（Constraints）

### 什麼是約束？

**比喻：遊戲規則**

就像籃球比賽有規則：
- 犯規會被吹哨
- 進球才能得分

資料庫的約束就是「資料的遊戲規則」：
- 違反規則 → 資料庫拒絕存入

### 常見約束

```sql
CREATE TABLE products (
    -- 主鍵約束：唯一且非空
    id INTEGER PRIMARY KEY,
    
    -- NOT NULL：不能是空白
    name TEXT NOT NULL,
    
    -- DEFAULT：沒給值時的預設值
    status TEXT DEFAULT 'active',
    
    -- UNIQUE：不能重複（但可以為空）
    sku TEXT UNIQUE,
    
    -- CHECK：自訂規則
    price INTEGER CHECK (price >= 0),
    
    -- 非空且有預設值
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

### 約束全覽表

| 約束 | 關鍵字 | 功能 |
|------|--------|------|
| 主鍵 | PRIMARY KEY | 唯一 + 非空 |
| 非空 | NOT NULL | 欄位必須有值 |
| 唯一 | UNIQUE | 值不能重複 |
| 預設值 | DEFAULT | 沒給值時自動填入 |
| 檢查 | CHECK | 自訂規則（例如 price > 0） |
| 外鍵 | FOREIGN KEY | 關聯其他表格 |

## 5️⃣ 實際操作（一步一步帶著做）

### 環境準備

打開 SQLite：

```bash
sqlite3 day3_practice.db
```

### 任務 1：創建「商品表」

```sql
CREATE TABLE products (
    id INTEGER PRIMARY KEY,           -- 商品ID（主鍵）
    name TEXT NOT NULL,              -- 商品名稱（必填）
    price DECIMAL(10,2),             -- 價格（精確到小數點）
    stock INTEGER DEFAULT 0,        -- 庫存（預設 0）
    category TEXT,                   -- 分類
    is_active BOOLEAN DEFAULT 1,     -- 是否上架（預設上架）
    created_at DATETIME               -- 建立時間
);
```

**解說：**
- `DECIMAL(10,2)` = 最多 10 位數，其中 2 位是小數（例如：99999999.99）
- `DEFAULT 0` = 沒特別指定就當作 0
- `BOOLEAN` = 1 代表 true（上架），0 代表 false（下架）

### 任務 2：查看表格結構

```sql
.schema products
```

會看到：
```sql
CREATE TABLE products (
    id INTEGER PRIMARY KEY,
    name TEXT NOT NULL,
    price DECIMAL(10,2),
    stock INTEGER DEFAULT 0,
    category TEXT,
    is_active BOOLEAN DEFAULT 1,
    created_at DATETIME
);
```

### 任務 3：插入資料

```sql
-- 插入完整資料
INSERT INTO products (name, price, stock, category)
VALUES ('iPhone 15', 29900, 100, '手機');

-- 只插入必要資料（其他用預設值）
INSERT INTO products (name, price, category)
VALUES ('小米手環', 899, '穿戴裝置');

-- 查詢看看
SELECT * FROM products;
```

### 任務 4：修改表格結構（ALTER）

```sql
-- 新增欄位
ALTER TABLE products ADD COLUMN description TEXT;

-- 修正：SQLite 不支援 ALTER TABLE 刪除欄位，
-- 如果要刪除欄位，需要重建表格（較複雜，初學者先跳過）
```

## 6️⃣ 常見錯誤/雷點提醒

### ⚠️ 新手常犯的錯

1. **TEXT 用太大**
   ```sql
   -- 錯誤 ❌
   name TEXT  -- 沒限制，可能浪費空間
   
   -- 正確 ✅
   name VARCHAR(100)  -- 最多 100 字元
   ```

2. **用 VARCHAR 存數字**
   ```sql
   -- 錯誤 ❌
   price VARCHAR(10)  -- 當成文字，無法做數學運算
   
   -- 正確 ✅
   price DECIMAL(10,2)  -- 可以做價格計算
   ```

3. **忘記設主鍵**
   ```sql
   -- 錯誤 ❌ （未來會很麻煩）
   CREATE TABLE orders (
       name TEXT,
       amount INTEGER
   );
   
   -- 正確 ✅
   CREATE TABLE orders (
       id INTEGER PRIMARY KEY,
       name TEXT,
       amount INTEGER
   );
   ```

4. **中文字段名（不建議）**
   ```sql
   -- 可以，但不建議 ❌
   CREATE TABLE 學生 (
       編號 INTEGER PRIMARY KEY,
       姓名 TEXT
   );
   
   -- 建議用英文 ✅
   CREATE TABLE students (
       id INTEGER PRIMARY KEY,
       name TEXT
   );
   ```

5. **BOOLEAN 用的很奇怪**
   ```sql
   -- 錯誤 ❌
   status TEXT  -- 用文字 'yes'/'no'，容易寫錯
   
   -- 正確 ✅
   is_active BOOLEAN  -- 用 1/0 或 TRUE/FALSE
   ```

## 💡 小知識

**資料庫命名慣例：**

```
表格命名：
- 單數還是複數？業界有兩派
  ✅ users（常見，表格是一堆 user）
  ✅ user（也常用，表格代表一種entity）
- 建議：用複數(users, products, orders)

欄位命名：
- 用底線分詞：user_name, created_at
- 不要用駝峰：userName ❌
- is_ 開頭表示布林值：is_active, is_deleted
```

**AUTOINCREMENT vs 不寫：**

```sql
-- SQLite 指定 AUTOINCREMENT（id 會一直往上加，不會重用被刪除的 id）
id INTEGER PRIMARY KEY AUTOINCREMENT

-- SQLite 不指定（id 會重用被刪除的 id，效能較好）
id INTEGER PRIMARY KEY
```

一般情況不需要 AUTOINCREMENT，效能會比較好！

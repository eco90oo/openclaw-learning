## 1️⃣ 基礎概念（生活比喻）

### 🔍 什麼是 Spy？

想像你是個**偵探**，你要確認一位員工（function）今天上班有沒有正常工作。

**傳統測試（Mock）** 就像：叫一個分身假扮員工，然後問「你有沒有被叫到？」—— 但你根本沒叫真的員工。

**Spy** 就像：你讓**真正的員工上班**，但你在旁邊偷偷觀察（錄影），事後檢查：
- ✅ 他被叫了幾次
- ✅ 他輸入什麼參數
- ✅ 他輸出了什麼結果
- ✅ 他有沒有偷懶沒做

**一句話概括**：Spy 是「讓真的代码運行，同時偷偷紀錄他的一舉一動」。

### 🎯 什麼是「行為驗證」？

傳統測試驗證**結果**（輸出對不對），行為驗證驗證**過程**（有沒有按照正確順序做）。

生活比喻：
- **結果驗證**：餐廳出餐，你檢查菜有沒有上齊
- **行為驗證**：廚師在監視器裡，你檢查他做菜的**步驟**對不對——先切菜、再熱鍋、而不是先熱鍋才發現沒切菜

### 2.1 內建 Spy（手動包裝）

最簡單的 Spy 就是：用 ` unittest.mock ` 的 `MagicMock` 包住一個真的函數：

```python
from unittest.mock import MagicMock

# 想像你有這樣一個真的服務
def send_email(to, subject, body):
    """這是你真正要測試的 email 發送函數"""
    print(f"發送郵件給 {to}，標題：{subject}")
    return True  # 正常運作會回傳 True

# 建立一個 Spy 包住它
email_spy = MagicMock(side_effect=send_email)

# 現在你可以「偷偷叫用它」
result = email_spy("user@example.com", "歡迎", "歡迎加入！")

# 驗證：這個間諜有沒有真的去執行原本的發送動作？
print(email_spy.called)        # 印出 True（確認有被叫用一次）
print(email_spy.call_count)   # 印出 1（確認只叫了一次）
print(email_spy.call_args)    # 印出 call('user@example.com', '歡迎', '歡迎加入！')
```

### 2.2 Spy 最核心的三件事：叫用次數、參數、回傳值

```python
# ─────────────────────────────────────────────
# 第一件事：檢查「叫用了幾次」
# ─────────────────────────────────────────────
print(email_spy.call_count)  # 1 → 代表被叫用了一次

# ─────────────────────────────────────────────
# 第二件事：檢查「叫用時傳了什麼參數」
# ─────────────────────────────────────────────
# call_args 是最近一次叫用的參數（類似tuple包裝）
args, kwargs = email_spy.call_args
print(args[0])  # 'user@example.com'
print(args[1])  # '歡迎'
print(args[2])  # '歡迎加入！'

# ─────────────────────────────────────────────
# 第三件事：檢查「每次叫用的完整歷史」
# ─────────────────────────────────────────────
# 呼叫兩次
email_spy("user2@test.com", "通知", "內容1")
email_spy("user3@test.com", "提醒", "內容2")

# 檢查歷史：所有叫用的參數清單
for i, call in enumerate(email_spy.call_args_list):
    print(f"第 {i+1} 次叫用：{call}")

# 輸出：
# 第 1 次叫用：call('user2@test.com', '通知', '內容1')
# 第 2 次叫用：call('user3@test.com', '提醒', '內容2')
```

### 2.3 一個完整的 Spy 實戰範例

情境：公司系統要寄「開工通知」給所有員工，你要測試：
- ✅ 有沒有對每個員工都寄信
- ✅ 寄的主旨是否正確

```python
from unittest.mock import MagicMock, call

def send_email(to, subject, body):
    """真正的 email 發送函數（被測試的對象）"""
    return True

def notify_all_employees(employees, email_func):
    """通知所有員工的主程式"""
    results = []
    for emp in employees:
        subject = f"【{emp['name']}】開工通知"
        body = f"親愛的 {emp['name']}，今天是開工日！"
        result = email_func(emp['email'], subject, body)
        results.append(result)
    return results

# === 建立 Spy 來監視 email_func ===
# 我們傳入的是「包著真的 send_email 的 Spy」
email_spy = MagicMock(side_effect=send_email)

# 員工清單
employees = [
    {"name": "小明", "email": "ming@example.com"},
    {"name": "大華", "email": "dahua@example.com"},
    {"name": "小美", "email": "mei@example.com"},
]

# 執行通知邏輯
results = notify_all_employees(employees, email_spy)

# === 行為驗證（核心！）===
# 1. 檢查 email 被叫用了 3 次（所有員工都有寄）
assert email_spy.call_count == 3, "應該要寄給 3 位員工"

# 2. 檢查寄給每一個人時的參數
expected_calls = [
    call("ming@example.com", "【小明】開工通知", "親愛的 小明，今天是開工日！"),
    call("dahua@example.com", "【大華】開工通知", "親愛的 大華，今天是開工日！"),
    call("mei@example.com", "【小美】開工通知", "親愛的 小美，今天是開工日！"),
]
email_spy.assert_has_calls(expected_calls)  # 確認叫用順序完全符合預期

print("✅ 行為驗證通過：所有員工都收到正確主旨的郵件")
```

### 2.4 更進階：Spy with return_value 和 side_effect 混合

有時候你不只想要「監視」，還想要「控制」它的行為：

```python
from unittest.mock import MagicMock

def risky_api_call(user_id):
    """會失敗的 API（真實環境可能網路不穩）"""
    raise ConnectionError("網路連線失敗")

# Spy 包住它，同時控制失敗行為
risky_spy = MagicMock(side_effect=risky_api_call)

# 在測試中，你可以指定「當叫用時，回傳特定值」而不真的呼叫
# 情境：第一次失敗，第二次成功（模擬重試機制）
def mock_with_retry(*args, **kwargs):
    """模擬重試後成功"""
    if not hasattr(mock_with_retry, 'count'):
        mock_with_retry.count = 0
    mock_with_retry.count += 1
    if mock_with_retry.count == 1:
        raise ConnectionError("第一次失敗")
    return {"status": "success", "data": "OK"}

api_spy = MagicMock(side_effect=mock_with_retry)

# 測試你的「重試邏輯」是否正確
def call_with_retry(api_func, max_retries=2):
    for attempt in range(max_retries):
        try:
            return api_func("user_001")
        except ConnectionError as e:
            if attempt == max_retries - 1:
                raise
            print(f"第 {attempt+1} 次失敗，準備重試...")

# 執行重試邏輯
result = call_with_retry(api_spy)

# 驗證：確認真的試了 2 次
assert api_spy.call_count == 2
print(f"✅ 重試機制正常：API 被叫用了 {api_spy.call_count} 次")
print(f"✅ 最終結果：{result}")
```

### 2.5 spy + pytest 整合實戰

```python
import pytest
from unittest.mock import MagicMock, call

def calculate_discount(user_tier, cart_total):
    """根據用戶等級計算折扣"""
    if user_tier == "gold":
        return cart_total * 0.8    # 金牌用戶打 8 折
    elif user_tier == "silver":
        return cart_total * 0.9    # 銀牌用戶打 9 折
    return cart_total              # 一般用戶不打折

def apply_final_price(cart_total, discount_func, tax_rate=1.05):
    """計算最終價格（包裝費用的地方）"""
    discounted = discount_func("gold", cart_total)
    return discounted * tax_rate   # 加上稅

# === 用 Spy 驗證「用了什麼折扣函數」 ===
def test_apply_final_price_uses_correct_discount():
    """測試：最終價格結算有沒有正確叫用折扣函數"""
    
    # 步驟 1：建立 Spy
    discount_spy = MagicMock(side_effect=calculate_discount)
    
    # 步驟 2：執行真正的邏輯，間諜在旁邊觀察
    result = apply_final_price(1000, discount_spy, tax_rate=1.1)
    
    # 步驟 3：行為驗證
    # 3a. 確認有叫用折扣函數
    assert discount_spy.called, "應該要叫用折扣函數"
    
    # 3b. 確認叫用的第一個參數是 "gold"（不是 silver 或普通用戶）
    call_args = discount_spy.call_args
    assert call_args[0][0] == "gold", "應該傳入金牌用戶等級"
    
    # 3c. 確認叫用的第二個參數是 1000（原價）
    assert call_args[0][1] == 1000, "應該傳入正確的購物車總額"
    
    # 3d. 驗證最終金額：1000 * 0.8 * 1.1 = 880
    assert result == 880, f"最終價格應該是 880，實際是 {result}"
    
    print(f"✅ Spy 驗證通過：叫用次數={discount_spy.call_count}, 結果={result}")
```

```python
# order_service.py
# 這是一個「訂單服務」系統，模擬真實場景

class InventorySystem:
    """庫存系統（真實邏輯）"""
    
    def check_stock(self, product_id):
        """檢查商品是否有庫存"""
        # 正常應該呼叫外部 API，這裡簡化模擬
        return True  # 假設都有庫存
    
    def reserve(self, product_id, quantity):
        """預留庫存"""
        print(f"📦 預留庫存：商品 {product_id} x {quantity}")
        return True


class PaymentGateway:
    """付款閘道（真實邏輯）"""
    
    def process(self, user_id, amount):
        """處理付款"""
        print(f"💳 處理付款：使用者 {user_id} 付 ${amount}")
        return {"status": "success", "transaction_id": "TX12345"}


class OrderService:
    """訂單服務（被測試的主體）"""
    
    def __init__(self, inventory, payment):
        # 依賴注入：傳入庫存系統和付款閘道
        self.inventory = inventory
        self.payment = payment
    
    def place_order(self, user_id, product_id, quantity, price):
        """下訂單的完整流程"""
        # 第一步：檢查庫存
        if not self.inventory.check_stock(product_id):
            return {"success": False, "reason": "庫存不足"}
        
        # 第二步：預留庫存
        reserve_ok = self.inventory.reserve(product_id, quantity)
        if not reserve_ok:
            return {"success": False, "reason": "預留失敗"}
        
        # 第三步：處理付款
        total = quantity * price
        payment_result = self.payment.process(user_id, total)
        
        if payment_result["status"] == "success":
            return {
                "success": True,
                "transaction_id": payment_result["transaction_id"],
                "total": total
            }
        else:
            return {"success": False, "reason": "付款失敗"}
```

### Step 3：建立 Spy 測試檔 `test_order_spy.py`

```python
# 用 Spy 來驗證「訂單流程」的行為是否正確

import sys
sys.path.insert(0, '.')

from unittest.mock import MagicMock, call
from order_service import OrderService, InventorySystem, PaymentGateway


def test_order_service_spy_behavior_verification():
    """
    情境：使用者小明下單買 2 個商品 A，單價 500
    
    我們要用 Spy 驗證：
    1. 庫存系統有沒有被正確叫用
    2. 付款閘道有沒有被正確叫用
    3. 叫用的參數是否完全符合預期
    """
    
    # === Step 1：建立 Spy 包住真實的庫存系統 ===
    # 這裡我們不 Mock 整個系統，而是 Spy 包住真實物件
    inventory_spy = MagicMock(
        wraps=InventorySystem()  # wraps=把真的物件包起來，間諜在外面觀察
    )
    
    # === Step 2：建立 Spy 包住真的付款閘道 ===
    payment_spy = MagicMock(wraps=PaymentGateway())
    
    # === Step 3：建立訂單服務，傳入間諜版本 ===
    order_service = OrderService(inventory_spy, payment_spy)
    
    # === Step 4：執行真正的下單邏輯 ===
    result = order_service.place_order(
        user_id="user_xiao_ming",
        product_id="product_A",
        quantity=2,
        price=500
    )
    
    # === Step 5：行為驗證（最核心的部分！）===
    
    # ✅ 驗證一：檢查庫存有没有被叫用
    assert inventory_spy.check_stock.called, "應該要叫用 check_stock"
    assert inventory_spy.reserve.called, "應該要叫用 reserve"
    
    # ✅ 驗證二：檢查叫用的具體參數
    # check_stock 應該被叫用一次，參數是 "product_A"
    check_stock_call = inventory_spy.check_stock.call_args
    assert check_stock_call[0][0] == "product_A", \
        f"check_stock 參數錯誤：預期 'product_A'，實際 {check_stock_call}"
    
    # reserve 應該被叫用一次，參數是 "product_A" 和 2
    reserve_call = inventory_spy.reserve.call_args
    assert reserve_call[0][0] == "product_A", "reserve 商品ID錯誤"
    assert reserve_call[0][1] == 2, "reserve 數量錯誤"
    
    # ✅ 驗證三：檢查付款閘道叫用參數
    # process 應該被叫用一次，參數是 "user_xiao_ming" 和 1000（2*500）
    payment_call = payment_spy.process.call_args
    assert payment_call[0][0] == "user_xiao_ming", "付款使用者錯誤"
    assert payment_call[0][1] == 1000, f"付款金額錯誤：預期 1000，實際 {payment_call[0][1]}"
    
    # ✅ 驗證四：檢查叫用順序（重要！）
    # 順序應該是：check_stock → reserve → process
    all_calls = inventory_spy.check_stock.call_args_list + \
                inventory_spy.reserve.call_args_list + \
                payment_spy.process.call_args_list
    
    # 我們用另一種方式驗證順序：確認沒有先付款再檢查庫存
    # 如果真的呼叫了，call_count 起碼會是 1
    assert inventory_spy.check_stock.call_count >= 1
    assert payment_spy.process.call_count == 1
    
    # ✅ 驗證五：檢查最終結果
    assert result["success"] == True, f"訂單應該成功：{result}"
    assert result["total"] == 1000, f"總金額應該是 1000：{result}"
    
    print("=" * 60)
    print("✅ 所有 Spy 行為驗證通過！")
    print(f"   - 庫存檢查叫用次數：{inventory_spy.check_stock.call_count}")
    print(f"   - 庫存預留叫用次數：{inventory_spy.reserve.call_count}")
    print(f"   - 付款處理叫用次數：{payment_spy.process.call_count}")
    print(f"   - 訂單結果：{result}")
    print("=" * 60)


# === 測試二：驗證「失敗流程」的行為 ===
def test_order_service_failed_flow_spy():
    """
    情境：庫存不足時，系統有沒有正確終止流程
    重要：當庫存不足時，不應該叫用付款！
    """
    
    # 製造一個「永遠沒庫存」的 Spy
    def mock_check_stock(product_id):
        return False  # 永遠回傳「沒庫存」
    
    inventory_mock = MagicMock()
    inventory_mock.check_stock.side_effect = mock_check_stock  # 用 side_effect 控制行為
    payment_spy = MagicMock(wraps=PaymentGateway())
    
    order_service = OrderService(inventory_mock, payment_spy)
    result = order_service.place_order("user_001", "prod_X", 1, 100)
    
    # ✅ 行為驗證：check_stock 有被叫用
    assert inventory_mock.check_stock.called, "應該叫用庫存檢查"
    
    # ✅ 行為驗證：但 reserve 不應該被叫用（因為庫存不足就停了）
    assert inventory_mock.reserve.call_count == 0, \
        "庫存不足時不應該叫用 reserve！"
    
    # ✅ 行為驗證：付款絕對不應該被叫用！
    assert payment_spy.process.call_count == 0, \

    # ✅ 驗證結果應該是失敗
    assert result["success"] == False
    assert result["reason"] == "庫存不足"
    
    print("=" * 60)
    print("✅ 失敗流程 Spy 驗證通過！")
    print(f"   - 庫存檢查叫用次數：{inventory_mock.check_stock.call_count}")
    print(f"   - 庫存預留叫用次數：{inventory_mock.reserve.call_count}")
    print(f"   - 付款處理叫用次數：{payment_spy.process.call_count}")
    print(f"   - 訂單結果：{result}")
    print("=" * 60)

    print("\n🔍 執行 Spy 行為驗證測試...\n")

    print()
    test_order_service_failed_flow_spy()
```

## 📚 今日重點摘要

```
┌─────────────────────────────────────────────────────────┐
│  Spy = 執行真的程式 + 監視它的所有行為                  │
├─────────────────────────────────────────────────────────┤
│  核心語法：                                             │
│  • spy = MagicMock(wraps=real_object)  → 包住真物件     │
│  • spy.call_count                  → 被叫用幾次         │
│  • spy.call_args                   → 最近一次叫用的參數 │
│  • spy.call_args_list              → 所有叫用的歷史     │
│  • spy.assert_has_calls([...])     → 驗證叫用清單       │
│  • spy.side_effect = fn            → 控制函數行為       │
├─────────────────────────────────────────────────────────┤
│  適用情境：                                             │
│  當你要驗證「過程」而非只驗證「結果」時                │
│  當你要確認某個函數有沒有被叫用、叫用幾次、參數是什麼 │
└─────────────────────────────────────────────────────────┘
```

> 💡 **Day 5 回顧**：Mock 用於「替代並控制行為」，Spy 用於「執行並監視行為」。兩者是互補的工具！

# 📚 【今日學習】資深 PM 教你從 0 到 1 開發產品（Day 1）

> 本系列為 30 天產品開發學習課程，從市場發現到產品推廣，帶你完整掌握產品經理的核心技能。

---

## 1️⃣ 基礎概念

### 什麼是產品經理（PM）？

產品經理（Product Manager，簡稱 PM）是一個產品的「負責人」或「CEO」。你要對產品的成敗負最終責任，協調開發、設計、行銷、營運等團隊，確保產品能解決使用者的真實問題，並且能夠商業化。

PM 不是「發指令」的人，而是「做決策」的人。你需要：
- 理解使用者想要什麼
- 決定要做什麼功能、先做什麼
- 確保團隊朝正確方向前進

### 從 0 到 1 開發產品是什麼意思？

「從 0 到 1」指的是從「沒有產品」到「創造出產品」的完整過程。這包括：
1. **發現問題** - 找到市場上未被滿足的需求
2. **驗證問題** - 確認這個問題真的有人願意花錢解決
3. **定義產品** - 決定產品要做什麼（PRD）
4. **打造 MVP** - 用最小成本做出最基本可用版本
5. **迭代優化** - 根據用戶反饋持續改進
6. **推廣市場** - 讓更多人知道並使用你的產品

### 產品開發的核心思維

**以用戶為中心**：永遠問自己「用戶真的需要這個嗎？」，而不是「這個功能好酷」。

**假設驗證**：產品開發是一連串的假設，你需要快速驗證這些假設是否正確。

**迭代優化**：沒有產品是一次就完美的，都是透過不斷測試、學習、改進循環前進。

## 2️⃣ 具體做法

### 階段一：產品發現與市場研究

#### 2.1 如何發現市場機會？

**觀察生活痛點**：想想自己或身邊人在日常生活中遇到什麼麻煩？什麼事情讓你覺得「如果有個東西能幫我解決這個該多好」？

**市場趨勢分析**：關注科技趨勢、產業報告、投資動向。例如：AI 興起、遠距工作常態化、老齡化社會等趨勢會創造哪些新需求？

**未被滿足的需求**：現有解決方案有哪些缺點？能否做得更好、更快、更便宜？

#### 2.2 市場研究怎麼做？

**市场规模（TAM/SAM/SOM）**：
- TAM（Total Addressable Market）：理論上最大市場規模
- SAM（Serviceable Addressable Market）：你能觸及的市場規模
- SOM（Serviceable Obtainable Market）：你能實際取得的市場份額

**研究方法**：
- **定量研究**：發放問卷、收集數據，了解市場有多大
- **定性研究**：訪談潛在用戶，了解他們的真實想法和痛點

### 階段二：競品分析

#### 2.3 為什麼要做競品分析？

了解競爭對手可以幫助你：
- 避免重複造輪子
- 找到差異化機會
- 學習成功/失敗經驗

#### 2.4 怎麼做競品分析？

**找到競爭對手**：
- 直接競爭者：提供類似產品的公司
- 間接競爭者：滿足同一需求的替代方案
- 潛在競爭者：可能進入這個市場的公司

**分析維度**：
| 分析維度 | 問題 |
|---------|------|
| 產品功能 | 對手有哪些功能？哪些做得比你好？ |
| 用戶體驗 | 對手的介面好嗎？使用流程順暢嗎？ |
| 定價策略 | 對手怎麼收費？你的價格優勢在哪？ |
| 市場定位 | 對手主打什麼價值主張？ |
| 獲客渠道 | 對手如何獲取用戶？ |

**競品分析工具**：
- 使用 SimilarWeb、App Annie 分析對手流量
- 實際註冊、使用對手產品
- 閱讀對手的用戶評價

### 階段三：定義產品

#### 2.5 PRD（產品需求文檔）怎麼寫？

PRD 是 PM 最重要的文件之一，它定義了「產品要做什麼」。

**PRD 基本結構**：

```
1. 產品概述
   - 產品名稱與版本
   - 產品目標與願景
   - 目標用戶是誰

2. 問題陳述
   - 用戶面臨什麼問題？
   - 這個問題有多嚴重？

3. 解決方案
   - 我們如何解決這個問題？
   - 產品的核心價值主張是什麼？

4. 功能需求
   - 每個功能詳細描述
   - 功能的使用流程
   - 邊界條件與例外處理

5. 非功能需求
   - 效能要求
   - 安全要求
   - 相容性要求

6. 衡量指標
   - 如何判斷這個功能成功了？
   - 關鍵指標（KPIs）有哪些？

7. 時間與優先級
   - 預計上線時間
   - 功能優先級排序
```

**寫 PRD 的小技巧**：
- 用「使用者故事」格式描述功能：作為[角色]，我希望[功能]，以便[價值]
- 包含具體的 wireframe 或草圖
- 讓開發、設計都能看懂

#### 2.6 MVP（最小可行產品）怎麼定義？

MVP 是「用最少資源做出來、能驗證假設的最基本產品」。

**如何定義 MVP**：
1. 列出所有你想要的功能
2. 問自己：「如果只能保留一個功能，會是哪個？」
3. 找到那個「沒有它產品就沒意義」的核心功能
4. 這就是你的 MVP

**常見錯誤**：
- ❌ MVP 不是「功能少一點的完整產品」
- ✅ MVP 是「能驗證核心假設的最精簡版本」

**案例**：Dropbox 當年的 MVP 只是一個介紹影片，沒有實際產品，但驗證了用戶對雲端檔案同步的需求。

### 階段四：功能優先級

#### 2.7 如何排序功能優先級？

**MoSCoW 方法**：
- Must have（必須有）：沒有這個功能產品無法運作
- Should have（應該有）：重要但非緊急
- Could have（可以有）：增加用戶滿意度
- Won't have（這次不做）：未來可能考慮

**Kano 模型**：
- 基本需求：做到沒感覺，沒做到很不爽
- 期望需求：做得越多越滿意
- 興奮需求：沒想過但超級驚喜

**RICE 分數**：
- Reach（觸及）：這個功能影響多少人？
- Impact（影響）：對用戶影響有多大？（3=巨大，2=高，1=中等，0.5=低）
- Confidence（信心）：你對這個估計有多大把握？
- Effort（工作量）：需要多少人/週？

計算：RICE Score = (Reach × Impact × Confidence) / Effort

## 3️⃣ 今日小任務

### 🎯 今天的實戰練習

**任務一：發現一個產品機會**
1. 想想你最近在使用什麼產品時感到不方便？
2. 記錄下來：這個痛點是什麼？現有解決方案有哪些缺點？

**任務二：做一個簡單的競品分析**
1. 選擇一個你有興趣的產品領域
2. 找出 2-3 個主要競爭對手
3. 用上面的維度表格分析他們

**任務三：練習寫一個使用者故事**
選擇一個功能，用以下格式寫出來：
```
作為[角色]，我希望[功能]，以便[價值]
```

例如：
> 作為「經常網購的人」，我希望「能夠一鍵查詢物流進度」，以便「不用每天打開好幾個 App 輸入單號」
